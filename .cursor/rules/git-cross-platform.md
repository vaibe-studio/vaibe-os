# Git: кросс-платформенная работа (Windows + Linux)

Цель: предотвратить проблемы синхронизации при совместной работе с разных ОС.

## Контекст

В репозитории работают участники с Windows (NTFS, case-insensitive) и Linux (ext4, case-sensitive). Имена файлов и директорий содержат кириллицу (UTF-8). Это создаёт два класса проблем: **конфликт регистра** и **конфликт кодировок**.

---

## Правило 1: Единый регистр в именах

**ЗАПРЕЩЕНО** создавать директории/файлы, отличающиеся только регистром:
- `vAIbe-OS/` и `vAIbe-os/` — на Linux это разные папки, на Windows — одна
- При переименовании директории (смена регистра) на Linux — старая запись остаётся в git-индексе и создаёт фантомный конфликт на Windows

**При переименовании с изменением регистра на Linux:**
1. `git mv old-name temp-name` (промежуточное имя)
2. `git mv temp-name New-Name` (финальное имя)
3. Коммит + push
4. На Windows: `git pull` получит чистое переименование

**При обнаружении конфликта на Windows (симптомы: `git status` показывает modified, `git add` не помогает, `git diff` пуст):**
— См. базу знаний: `Проекты/vAIbe-OS/База знаний/Git кросс-платформенная синхронизация.md`

## Правило 2: Кириллица в путях через консоль Windows

Консоль Windows (PowerShell, cmd, Git Bash) некорректно передаёт кириллические пути в git из-за несовпадения кодировок (CP866 / Windows-1251 vs UTF-8).

**Симптомы**: `git hash-object "Проекты/..."` → `fatal: could not open 'РџСЂРѕРµРєС‚С‹/...'`

**Решение**: Для операций с кириллическими путями использовать **Python-обёртку**:

```python
import subprocess
subprocess.run(['git', 'add', '--', 'Проекты/vAIbe-OS/'], capture_output=True)
```

Python корректно передаёт UTF-8 пути в git через системные API.

---

## Правило 3: Настройки git для кросс-платформенности

Рекомендуемые настройки в `.gitattributes` (корень репозитория):

```
* text=auto
*.md text eol=lf
*.png binary
*.jpg binary
```

Это гарантирует:
- Текстовые файлы всегда хранятся с LF в репозитории
- Бинарные файлы не конвертируются
- Поведение одинаково на всех ОС

## Правило 4: Диагностика фантомных изменений

Если `git status` показывает modified файлы, которые невозможно застейджить:

1. **Проверить дубли регистра** в индексе:
   ```python
   import subprocess
   result = subprocess.run(['git', 'ls-files', '--stage', '-z'], capture_output=True)
   entries = result.stdout.split(b'\0')
   paths = [e.split(b'\t', 1)[1] for e in entries if b'\t' in e]
   # Найти пары, отличающиеся только регистром
   lower_map = {}
   for p in paths:
       key = p.lower()
       if key in lower_map and lower_map[key] != p:
           print(f"КОНФЛИКТ: {lower_map[key]} vs {p}")
       lower_map[key] = p
   ```

2. **Удалить старые записи** через `git update-index --force-remove`
3. **Пере-добавить файлы** через `git add`

## Правило 5: Изоляция личных данных в общих репозиториях

Если репозиторий используется совместно, личные проекты и данные **НЕ ДОЛЖНЫ** попадать в общий индекс.

**Категоризация файлов перед коммитом:**

| Категория | Примеры | Действие |
|-----------|---------|----------|
| Системные | `.cursor/rules/`, `.cursor/commands/`, `Роли/` | Коммитить |
| Рабочие проекты | `Проекты/Videozo/`, `Проекты/ЕПК ФРИИ/` | Коммитить |
| Личные проекты | `Проекты/Музыкальный альбом/`, `База знаний/Творчество/` | Исключить через `.gitignore` |
| Временные импорты | `Инбокс/` | Исключить через `.gitignore` |

**Формат `.gitignore` для личных данных:**
```
# Private local project materials
/Проекты/Название личного проекта/
/База знаний/Личная тема/
/Инбокс/
```

**Протокол коммита:**
1. Проверить `.gitignore` — все личные директории исключены
2. `git add -u` — застейджить модификации/удаления отслеживаемых файлов
3. Селективно добавить новые файлы (не `git add -A`, не `git add .`)
4. `git status --porcelain` — убедиться, что личные данные не в staging area
5. Коммит

## Правило 6: Селективный staging кириллических путей

PowerShell искажает кириллические пути при передаче в git. Для добавления файлов с кириллическими путями использовать `--pathspec-from-file`:

1. Создать файл `.git/paths-to-add.txt` (UTF-8) со списком путей — по одному на строку
2. Выполнить `git add --pathspec-from-file=.git/paths-to-add.txt`

**Пример** `.git/paths-to-add.txt`:
```
Проекты/vAIbe-OS/Задачи/013-Новая задача/task.md
Роли/Новая роль/README.md
```

Этот подход обходит проблему кодировки PowerShell и надёжно работает с любыми Unicode-путями.

## Правило 7: Очистка garbled-директорий (double-encoded UTF-8)

При работе на Windows могут возникать директории с "мохибаке" — двойной кодировкой UTF-8 (например, `РџСЂРѕРµРєС‚С‹` вместо `Проекты`). Это артефакты операций с кириллическими путями через консоль с неправильной кодировкой.

**Диагностика**: `git ls-files --others --exclude-standard --directory` — garbled-директории видны как пути с Р, С, Рѕ, Рё и подобными символами.

**Очистка** через Python (корректно работает с Unicode на Windows):
```python
import os, shutil
root = r'c:\path\to\repo'
known = {'Проекты', 'Роли', 'База знаний', ...}
for d in os.listdir(root):
    full = os.path.join(root, d)
    if os.path.isdir(full) and d not in known:
        if any(c in d for c in ['Р', 'С']):
            shutil.rmtree(full)  # garbled directory
```

**Профилактика**: Всегда использовать Python-обёртку или `--pathspec-from-file` (Правило 6) для git-операций с кириллическими путями.
