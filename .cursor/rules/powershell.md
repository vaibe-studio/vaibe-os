# PowerShell: совместимость команд на Windows

Цель: избежать повторяющихся ошибок при выполнении shell-команд в среде PowerShell (Windows).

## Определение среды

Если в `user_info` указано `Shell: powershell` или `OS Version: win32` — все команды выполняются в PowerShell. Применяй правила ниже.

## Запрещённые конструкции

| Конструкция | Почему ломается | Что использовать |
|-------------|----------------|-----------------|
| `<<'EOF' ... EOF` (heredoc) | PowerShell не поддерживает bash heredocs | Записать текст в файл → использовать файл |
| `&&` (цепочка команд) | Не работает в PowerShell < 7 | Выполнять команды **отдельными вызовами Shell** |
| `-m "длинный русский текст с (скобками)"` | PS интерпретирует `(...)` как subexpression, кириллица может ломать парсер | Использовать `git commit -F <файл>` |
| `$(command)` внутри строк | PS подставляет свои переменные / выражения | Избегать; использовать промежуточные файлы |

## Git commit с русским текстом (рекомендуемый паттерн)

### Шаг 1: записать сообщение в файл

Использовать инструмент **Write** (не echo/heredoc) для создания временного файла:

```
Путь: .commit_msg.tmp
Содержимое: текст коммит-сообщения (любой длины, любые символы)
```

### Шаг 2: staging (отдельный вызов Shell)

```powershell
git add "путь/к/файлам/"
```

### Шаг 3: commit (отдельный вызов Shell)

```powershell
git commit -F .commit_msg.tmp
```

### Шаг 4: проверка + очистка (отдельный вызов Shell)

```powershell
git status
```

После успешного коммита удалить `.commit_msg.tmp` через инструмент **Delete**.

## Кириллические пути: файловые операции (КРИТИЧЕСКИ ВАЖНО)

**Корневая причина**: Shell tool сохраняет команды во временный `.ps1`-файл. Кириллические символы в путях искажаются при записи (double-encoding / Mojibake). Это затрагивает **все** shell-команды: `copy`, `Copy-Item`, `move`, `Move-Item`, `python shutil.copy(прямой_текст)`, `cmd /c copy` и т.д.

### Что работает и что НЕ работает

| Операция | Кириллица прямым текстом | Unicode-эскейпы в Python |
|----------|:------------------------:|:------------------------:|
| `Copy-Item` / `copy` | ❌ | — |
| `Move-Item` / `move` | ❌ | — |
| `cmd /c copy` | ❌ | — |
| `python shutil.copy2()` | ❌ | ✅ `\uXXXX` |
| `python shutil.move()` | ❌ | ✅ `\uXXXX` |
| `python os.makedirs()` | ❌ | ✅ `\uXXXX` |
| `mkdir` (PowerShell) | ✅ (создание) | — |
| `git add` | ❌ (часто) | ✅ через `subprocess` |
| Cursor Read / Write / Delete / Glob | ✅ (нативно) | — |

### Выбор способа по типу операции

#### Копирование / перемещение файлов → Python shutil (ПРИОРИТЕТ 1)

> **КРИТИЧНО**: НЕ использовать Read + Write для копирования/перемещения файлов!
> Read + Write заставляет AI перегонять весь контент через себя — для файла в 1000+ строк это минуты ожидания, расход токенов и риск случайных изменений.

**Копирование файла** (1 секунда, файл копируется напрямую на диске):

```python
python -c "import shutil, os; shutil.copy2(os.path.join('c:\\vaibe-os-adr\\ai-os', '\u0418\u043d\u0431\u043e\u043a\u0441', 'file.md'), os.path.join('c:\\vaibe-os-adr\\ai-os', '\u041f\u0440\u043e\u0435\u043a\u0442\u044b', 'Videozo', '\u0412\u0441\u0442\u0440\u0435\u0447\u0438', 'dest.md'))"
```

**Перемещение файла** (1 секунда):

```python
python -c "import shutil, os; shutil.move(os.path.join('c:\\vaibe-os-adr\\ai-os', '\u0418\u043d\u0431\u043e\u043a\u0441', 'file.md'), os.path.join('c:\\vaibe-os-adr\\ai-os', '\u041f\u0440\u043e\u0435\u043a\u0442\u044b', 'dest.md'))"
```

**Создание директории**:

```python
python -c "import os; os.makedirs(os.path.join('c:\\vaibe-os-adr\\ai-os', '\u041f\u0440\u043e\u0435\u043a\u0442\u044b', 'Videozo', '\u0412\u0441\u0442\u0440\u0435\u0447\u0438', 'New Folder'), exist_ok=True)"
```

**Git add**:

```python
python -c "import subprocess; subprocess.run(['git', 'add', '--', '\u041f\u0440\u043e\u0435\u043a\u0442\u044b/Videozo/README.md'])"
```

#### Создание новых файлов, редактирование, чтение содержимого → Cursor-инструменты

Cursor-инструменты работают нативно с кириллицей и идеальны когда нужен доступ к **содержимому**:

- **Создание нового файла** (summary.md, tasks.md): `Write`
- **Чтение для анализа**: `Read`
- **Редактирование**: `StrReplace`
- **Удаление**: `Delete`
- **Поиск**: `Glob` / `Grep`

> **Правило**: `Read` + `Write` для «копирования» — только если файл МАЛЕНЬКИЙ (< 50 строк) и нужно его прочитать в том же flow. Для остального — Python shutil.

#### НИКОГДА не использовать

- `Copy-Item` / `copy` / `Move-Item` / `move` с кириллическими путями в Shell
- `cmd /c` с кириллическими путями
- Попытки «починить» encoding (`chcp 65001`, `$OutputEncoding`) — НЕ помогают
- `Read` + `Write` для копирования больших файлов (> 50 строк) — медленно, дорого, рискованно

## Общие правила для Shell-вызовов на Windows

1. **Одна команда = один вызов Shell.** Не цепочить через `&&`, `;`, `|` с побочными эффектами.
2. **Кириллица в путях** — НИКОГДА не передавать прямым текстом в shell-команды для файловых операций. Использовать Cursor-инструменты (Read/Write/Delete) или Python с Unicode-эскейпами (см. раздел выше).
3. **Staging теряется между вызовами Shell** — проверять через `git status` перед коммитом; если потерялся — повторить `git add`.
4. **Не использовать bash-измы**: `export`, `source`, `cat <<`, `$(...)`, `${}`.
5. **Кириллические пути ломаются в git** — консоль Windows (CP866) искажает UTF-8 пути. Если `git add`/`git hash-object` не находит файл с кириллицей, использовать Python: `subprocess.run(['git', 'add', '--', 'Путь/...'])` с Unicode-эскейпами.
6. **Фантомные modified файлы** — если `git status` показывает modified, `git diff` пуст и `git add` не помогает — это конфликт регистра в именах (Windows case-insensitive, Linux case-sensitive). См. `.cursor/rules/git-cross-platform.md`.

## Удаление директорий в PowerShell (частая ошибка)

- `rmdir /s /q ...` — это **cmd-стиль**. В PowerShell `rmdir` — алиас на `Remove-Item`, и параметры вида `/s /q` ломают команду.
- Использовать:

```powershell
Remove-Item -Recurse -Force ".tmp\\some-folder"
```

> Примечание: для путей с кириллицей действуют правила выше (предпочитать Cursor-инструменты или Python с Unicode-эскейпами).

## Экспорт “готового результата” в PDF (рекомендуемый путь)

Если пользователь просит артефакт “в формате PDF”, **не использовать** headless-браузер / внешние движки.

В репозитории уже есть конвертер:
- `tools/markdown_to_pdf/` (см. `tools/TOOLS_INDEX.md`)

Рецепт:
1) Создать/обновить Markdown-файл через Cursor-инструменты (в папке `results/` задачи).
2) Сгенерировать PDF через Python:

```powershell
python -m tools.markdown_to_pdf "input.md" -o "output.pdf"
```

Плюсы:
- не требует `pandoc`/LaTeX/wkhtmltopdf
- корректная кириллица (через системные Windows-шрифты)
