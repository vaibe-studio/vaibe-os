# Алгоритм обработки файлов из Инбокса

## Обзор
Алгоритм обработки файлов из Инбокса для импорта в структуру директорий vAIbe-os. Команда `/inbox-check` просматривает файлы в папке `Инбокс/`, **интерактивно запрашивает контекст у пользователя для каждого файла/группы** и производит перенос файлов в соответствующие директории согласно структуре ОС.

**ВАЖНО**: Строго соблюдайте структуру файловой системы согласно правилам в `.cursor/rules/structure.md`.

---

## Формат вызова
- `/inbox-check`

## Входы (источники данных)
- `Инбокс/` — файлы и папки для разбора
- `.cursor/rules/structure.md` — правила структуры
- (опционально) `tools/pdf_to_markdown/` — для извлечения текста из PDF

## Выходы (артефакты)
- Перемещённые/переименованные файлы в `Проекты/`, `Роли/`, `База знаний/` (по решению пользователя или в делегированном режиме)
- (опционально) извлечённый Markdown рядом с PDF: `{Название} (извлечено).md`
- Лог решений (в сообщении агентом) в быстром/делегированном режиме

---

## КРИТИЧЕСКИЕ ПРАВИЛА ИНТЕРАКТИВНОСТИ

> **ЗАПРЕЩЕНО**: Импортировать файлы без явного подтверждения пользователя.
> **ЗАПРЕЩЕНО**: Принимать финальные решения о целевой директории/категории **без подтверждения** пользователя (кроме явно включённого делегированного режима).
> **ЗАПРЕЩЕНО**: Пропускать шаг запроса контекста, даже если контекст "очевиден".
> **ОБЯЗАТЕЛЬНО**: После каждого вопроса — ОСТАНОВИТЬСЯ и ДОЖДАТЬСЯ ответа пользователя.
> **ОБЯЗАТЕЛЬНО**: Использовать инструмент AskQuestion для структурированных вопросов.

---

## Политика принятия решений (чтобы не было “двойных стандартов”)

- **По умолчанию (строгий/пошаговый/групповой режимы)**:
  - агент **формулирует гипотезы** (куда класть, как назвать, надо ли конвертировать)
  - агент предлагает **2–4 варианта** (если есть неопределённость) + обоснование
  - пользователь **подтверждает/правит** решение
  - только после подтверждения агент создаёт папки/перемещает файлы/переименовывает

- **Быстрый режим**:
  - агент готовит **единый план импорта** на все элементы
  - пользователь правит план
  - одно подтверждение на выполнение плана

- **Делегированный режим (только по явному разрешению пользователя)**:
  - пользователь заранее разрешает: “можешь сам раскладывать по правилам”
  - агент принимает решения **в рамках правил структуры и именования**
  - агент обязан вести **лог решений** (что, куда, почему), и в конце показать сводку
  - делегированный режим **не отменяет** запрет на импорт без подтверждения: подтверждение переносится “на конец” пакета (одним решением)

## Алгоритм

### 1. **Инициализация обработки**
   - Пользователь использует команду `/inbox-check`
   - Агент сканирует папку `Инбокс/` на наличие файлов
   - Если Инбокс пуст → информирование пользователя и завершение
   - Если есть файлы → переход к шагу 2

### 2. **Анализ файлов в Инбоксе**
   - Агент получает список всех файлов в `Инбокс/`
   - **Исключает из обработки служебные файлы:**
     - `.gitkeep` - служебный файл для Git
     - `README.md` - документация папки
   - Для каждого файла определяет:
     - Имя файла
     - Расширение и тип файла
     - Размер файла
     - Дату создания/изменения
   - Создает список файлов для обработки (исключая служебные файлы)
   - **Группирует связанные файлы** (например, все встречи, все задачи одного проекта)
   - **Обрабатывает папки как логические группы**:
     - Если в `Инбокс/` есть подпапка (например, папка проекта) — сначала обсуждаем её как “единицу”, затем разбираем содержимое
     - Допускается рекурсивный просмотр содержимого папки для первичной классификации (без принятия решений за пользователя)
     - В групповом режиме: сначала подтверждаем общий контекст (проект/категория), затем пофайлово уточняем спорные элементы

### 3. **Обработка файлов (ИНТЕРАКТИВНЫЙ ЦИКЛ)**

> **СТОП-ПРАВИЛО**: После каждого вопроса агент ОБЯЗАН остановиться и дождаться ответа пользователя. Переход к следующему шагу ТОЛЬКО после получения ответа.

   **Важно:** подвергать критическому анализу содержимое файлов; интерактивно вовлекай пользователя для выяснения ключевых вопросов и проверки фактов.

   Для каждого файла или логической группы файлов:

   #### 3.1. **Анализ содержимого файла**
   - Агент пытается прочитать содержимое файла (если возможно)
   - Если обрабатывается **папка** (логическая группа) — агент:
     - Показывает список файлов внутри (1 уровень или рекурсивно — по ситуации)
     - Предлагает разбиение на “основное” и “вторичное”
     - Формулирует гипотезу по общему контексту группы (без принятия решения)
   - Анализирует содержимое для определения типа материала:
     - Транскрипт встречи
     - Документация проекта
     - Материал для базы знаний роли
     - Пользовательский материал
     - Скриншот/изображение
     - Другое
   - Формулирует **гипотезу** о назначении файла (но НЕ принимает решение)

   #### 3.2. **ОБЯЗАТЕЛЬНЫЙ запрос контекста у пользователя**
   
   > ⚠️ **ТОЧКА ОСТАНОВА**: Агент ОБЯЗАН задать вопросы и ДОЖДАТЬСЯ ответа. НЕ продолжать без ответа пользователя.
   
   **Агент показывает пользователю:**
   - Имя файла (или список файлов в группе)
   - Тип файла
   - Предварительный анализ содержимого (краткое резюме)
   - **Свою гипотезу** о назначении (для проверки пользователем)
   
   **Агент ЗАДАЁТ ВОПРОСЫ (используя AskQuestion или текстовый формат):**
   
   ```
   📁 Файл: {имя файла}
   📋 Мой анализ: {краткое описание содержимого}
   💡 Моя гипотеза: {предположение о назначении}
   
   Вопросы:
   1. К какому проекту относится? (или "новый проект", или "не относится к проекту")
   2. Тип материала: встреча / задача / исходный материал / база знаний / роль / другое
   3. Предлагаемое название: {моё предложение} — согласны или предложите своё?
   4. Дополнительный контекст? (опционально)
   ```
   
   **Если материал классифицирован как "база знаний", агент ОБЯЗАН дополнительно:**
   
   ```
   🎯 Анализ универсальности материала:
   - Универсальность: {высокая/средняя/низкая} — {обоснование}
   - Проектный контекст: {есть/нет} — {что именно}
   - Переиспользуемость: {да/нет} — {в каких случаях}
   
   📍 Моё предложение по размещению:
   - {вариант: "только общая БЗ" / "только проектная БЗ" / "дублировать в обе"}
   - Обоснование: {почему именно так}
   
   Если дублирование:
   - Общая БЗ: `База знаний/{Название}.md`
   - Проектная БЗ: `Проекты/{Проект}/База знаний/{Название} ({Источник}).md`
   
   Согласны с предложением? (да / изменить)
   ```
   
   **→ СТОП. Ожидание ответа пользователя.**

   #### 3.3. **Определение целевой директории**
   **ТОЛЬКО ПОСЛЕ получения ответа пользователя** агент определяет целевую директорию:
   
   - **Если материал относится к проекту:**
     - `Проекты/{Название проекта}/` - для общей документации проекта
     - `Проекты/{Название проекта}/Задачи/{NUM}-{TITLE}/` - для материалов конкретной задачи
     - `Проекты/{Название проекта}/Встречи/Встреча {N}. {DATE:DD.MM.YYYY} - {TITLE}/` - для материалов встречи
     - `Проекты/{Название проекта}/Исходные материалы/` - для исходных материалов проекта
     - `Проекты/{Название проекта}/База знаний/` - для базы знаний проекта
   
   - **Если материал относится к роли:**
     - `Роли/{Название роли}/База знаний/` - для материалов базы знаний роли
   
   - **Если это пользовательский материал:**
     - `База знаний/` - для пользовательских материалов
     - Возможно создание подпапок по темам: `База знаний/{Тема}/`
   
   - **Если это скриншот/изображение:**
     - Определяется на основе контекста (проект, задача, база знаний)

   #### 3.4. **Обработка и переименование файла**
   - Агент создает целевые директории, если их нет
   - Переименовывает файл согласно правилам именования ОС:
     - Использование русского языка
     - Заглавные буквы для первых букв слов
     - Удаление дат из названия (если они есть в оригинале)
     - Приведение к единому формату
   - Если файл текстовый и требует конвертации → конвертация в Markdown (если необходимо)
   - Если файл — **PDF**, по умолчанию предлагаем 2 артефакта:
     - Оригинал PDF в `Проекты/{Проект}/Исходные материалы/...`
     - Рядом извлечённый текст в Markdown: `{Название} (извлечено).md`
       - Инструмент: `tools/pdf_to_markdown/` (без OCR, если текст извлекается; OCR — только по необходимости)

   #### 3.4.1. **Осознанное откладывание**
   Если элемент нельзя корректно разложить (не хватает контекста / спорная категория / “другой поток”):
   - Явно фиксируем причину (“почему откладываем”)
   - Перемещаем элемент в `Инбокс/Отложено/` (или другую согласованную очередь)
   - Возвращаемся к нему отдельной сессией `/inbox-check` (как самостоятельной группой)

   #### 3.5. **Подтверждение перед импортом**
   
   > ⚠️ **ТОЧКА ОСТАНОВА**: Перед выполнением импорта показать план и запросить подтверждение.
   
   **Агент показывает план импорта:**
   ```
   ✅ План импорта:
   - Исходный файл: {путь}
   - Целевая директория: {путь}
   - Новое имя: {имя}
   - Дополнительные действия: {если есть}
   
   Выполнить импорт? (да/нет/изменить)
   ```
   
   **→ СТОП. Ожидание подтверждения пользователя.**

   #### 3.6. **Импорт файла**
   **ТОЛЬКО ПОСЛЕ подтверждения** агент выполняет:
   - Перемещает файл из `Инбокс/` в целевую директорию
   - Если файл был переименован → сохраняет новое имя
   - Создает или обновляет связанные файлы при необходимости:
     - Обновление README.md проекта (если добавлен новый материал)
     - Обновление task.md задачи (если добавлен материал к задаче)

   #### 3.7. **Документирование импорта**
   - Агент фиксирует информацию об импорте:
     - Исходное имя файла
     - Новое имя файла
     - Целевая директория
     - Дата импорта
     - Контекст от пользователя
   
   #### 3.8. **Переход к следующему файлу**
   - Агент спрашивает: "Продолжить обработку следующего файла?"
   - **→ СТОП. Ожидание ответа.** (или продолжить, если пользователь ранее выбрал "обработать все")

### 4. **Завершение обработки**
   - После обработки всех файлов агент:
     - Показывает сводку импортированных файлов
     - Показывает список файлов, которые не удалось обработать (если есть)
     - Предлагает очистить Инбокс от обработанных файлов
     - Информирует о следующем шаге (если необходимо)

## Режимы обработки

В начале обработки агент ОБЯЗАН спросить у пользователя предпочтительный режим:

### Режим 1: Пошаговый (по умолчанию)
- Каждый файл обрабатывается отдельно
- Полный цикл вопросов для каждого файла
- Максимальный контроль пользователя

### Режим 2: Групповой
- Связанные файлы группируются (все встречи, все задачи проекта)
- Вопросы задаются для группы целиком
- Подтверждение для всей группы

### Режим 3: Быстрый (только для опытных)
- Агент предлагает план для ВСЕХ файлов сразу
- Пользователь корректирует план
- Одно подтверждение на весь импорт
- **ВАЖНО**: Даже в быстром режиме ОБЯЗАТЕЛЬНО показать план и получить подтверждение

### Режим 4: Делегированный (только по явному разрешению)
- Пользователь явно говорит: “раскладывай сам по правилам”
- Агент принимает решения о целевых директориях/именовании/конвертации
- Одно подтверждение “применить план” перед фактическим импортом (как в быстром)
- **ВАЖНО**: агент обязан показать лог принятых решений (что/куда/почему)

## Примеры

### Пример 1: Транскрипт встречи по проекту
**Исходный файл**: `Инбокс/Встреча по проекту ЕПК 20.01.2026.txt`

1. Анализ: текстовый файл, содержит транскрипт встречи
2. Запрос контекста:
   - Проект: "ЕПК"
   - Тип: транскрипт встречи
   - Номер встречи: 1
   - Дата: 20.01.2026
   - Название: "Обсуждение архитектуры"
   - Предлагаемое название: "transcript.md"
3. Импорт: `Проекты/ЕПК/Встречи/Встреча 1. 20.01.2026 - Обсуждение архитектуры/transcript.md`
4. Обновление: `Проекты/ЕПК/README.md` (добавлена ссылка на встречу)

### Пример 2: Материал для базы знаний роли
**Исходный файл**: `Инбокс/Лучшие практики разработки API.docx`

1. Анализ: документ Word, содержит материалы по разработке API
2. Запрос контекста:
   - Роль: "Разработчик"
   - Тип: база знаний роли
   - Предлагаемое название: "Лучшие практики разработки API.md"
3. Импорт: `Роли/Разработчик/База знаний/Лучшие практики разработки API.md`
4. Конвертация: DOCX → Markdown (если необходимо)

### Пример 3: Пользовательский материал
**Исходный файл**: `Инбокс/Заметки по Git.txt`

1. Анализ: текстовый файл, содержит заметки
2. Запрос контекста:

   - Предлагаемое название: "Команды Git.md"

   - Проект: "ЕПК"
   - Тип: исходный материал проекта
   - Предлагаемое название: "Техническое задание.md"
3. Импорт: `Проекты/ЕПК/Исходные материалы/Техническое задание.md`

### Пример 6: Скриншот интерфейса

2. Запрос контекста:
   - Проект: "AI Операционная Система"
   - Задача: "001-Проектирование системы"
   - Тип: документация задачи
   - Предлагаемое название: "Скриншот интерфейса.png"

### Пример 7: Методологический материал от трекера (дублирование в обе БЗ)
**Исходный файл**: `Инбокс/Презентация по ценностному предложению от ФРИИ.pdf`

1. Анализ: PDF с методологией построения ценностного предложения
2. Оценка универсальности:
   - Универсальность: **высокая** — фреймворк применим к любому B2B/B2C продукту
   - Проектный контекст: **есть** — получено от трекера ФРИИ в рамках акселератора
   - Переиспользуемость: **да** — можно использовать в других проектах
3. Решение: **дублировать в обе БЗ**
4. Импорт:
   - Общая БЗ: `База знаний/Ценностное предложение.md` (универсальная версия)
   - Проектная БЗ: `Проекты/ЕПК ФРИИ/База знаний/Ценностное предложение (ФРИИ Спринт).md` (с контекстом)
   - Оригинал: `Проекты/ЕПК ФРИИ/Исходные материалы/Ценностное предложение ФРИИ/`
5. Различия версий:
   - Общая: чистая методология без упоминания проекта
   - Проектная: + контекст "получено от трекера", + ссылки на связанные задачи

### Пример 8: Проектно-специфичный материал (только проектная БЗ)
**Исходный файл**: `Инбокс/Архитектура решения ЕПК.md`

1. Анализ: документ с описанием архитектуры конкретного продукта
2. Оценка универсальности:
   - Универсальность: **низкая** — описывает конкретное решение для ЕПК
   - Проектный контекст: **да** — специфичен для проекта
   - Переиспользуемость: **нет** — не применим к другим проектам
3. Решение: **только проектная БЗ**
4. Импорт: `Проекты/ЕПК ФРИИ/База знаний/Архитектура решения.md`

- Предложение вариантов на основе анализа содержимого

## АНТИПАТТЕРНЫ (чего НЕ делать)

### ❌ НЕПРАВИЛЬНО: Массовый импорт без вопросов
```
Агент: "Я вижу 15 файлов в Инбоксе. Все относятся к проекту X, импортирую..."
[выполняет импорт]
```
**Проблема**: Пользователь не мог скорректировать названия, исключить файлы, указать другой проект.

### ❌ НЕПРАВИЛЬНО: Один общий вопрос на все файлы
```
Агент: "Найдено 15 файлов. Все относятся к проекту ЕПК ФРИИ?"
Пользователь: "Да"
[агент сразу импортирует всё]
```
**Проблема**: Не было возможности уточнить детали для каждого файла/группы.

### ❌ НЕПРАВИЛЬНО: Предположение = решение
```
Агент: "Файл 'встреча.txt' — это транскрипт встречи, импортирую в Встречи/..."
```
**Проблема**: Гипотеза не была подтверждена пользователем.

### ✅ ПРАВИЛЬНО: Интерактивный цикл
```
Агент: "📁 Найдено 3 папки встреч в Инбоксе.
       
       Встреча 1. 16.01.2026:
       - transcript.txt (транскрипт)
       - meeting_summary.md (саммари)
       
       💡 Моя гипотеза: относится к проекту ЕПК ФРИИ
       
       Вопросы:
       1. Проект: ЕПК ФРИИ — верно?
       2. Название встречи: предлагаю 'Знакомство с акселератором' — согласны?
       3. Обработать все 3 встречи с таким же проектом?"

Пользователь: "1. Да. 2. Лучше 'Первая встреча с трекером'. 3. Да, но спроси про названия остальных"

Агент: [применяет ответы, спрашивает про следующую встречу]
```

## Разделение общей и проектной базы знаний

### Критерии для ОБЩЕЙ базы знаний (`База знаний/`)

Материал относится к **общей базе знаний**, если он:

1. **Универсален** — применим к любому проекту, не привязан к конкретному контексту
2. **Методологический** — описывает общие подходы, фреймворки, best practices
3. **Переиспользуемый** — будет полезен в других проектах без адаптации
4. **Объективный** — содержит факты/знания, а не проектные решения

**Примеры для общей БЗ:**
- Методология построения ценностного предложения (универсальный фреймворк)
- Структура welcome-презентации (шаблон для любого продукта)
- Чек-листы и шаблоны общего назначения
- Справочные материалы по технологиям

### Критерии для ПРОЕКТНОЙ базы знаний (`Проекты/{Проект}/База знаний/`)

Материал относится к **проектной базе знаний**, если он:

1. **Контекстный** — содержит информацию специфичную для проекта
2. **Адаптированный** — модифицирован под нужды проекта
3. **Исторический** — важен источник/контекст получения материала
4. **Связанный** — ссылается на другие артефакты проекта

**Примеры для проектной БЗ:**
- Материалы от трекера/ментора с контекстом проекта
- Адаптированные методологии с примерами из проекта
- Решения, принятые в рамках проекта

### Правила дублирования материалов

Если материал **одновременно универсален И получен в контексте проекта**:

1. **В общую БЗ** — "чистая" версия без проектного контекста
   - Имя файла: универсальное (`Ценностное предложение.md`)
   - Содержимое: только методология, примеры обезличены
   
2. **В проектную БЗ** — версия с контекстом проекта
   - Имя файла: с указанием источника (`Ценностное предложение (ФРИИ Спринт).md`)
   - Содержимое: + контекст получения, + ссылки на связанные задачи/материалы

### Правила именования файлов

| Расположение | Формат имени | Пример |
|--------------|--------------|--------|
| Общая БЗ | `{Тема}.md` | `Welcome-презентация.md` |
| Проектная БЗ | `{Тема} ({Источник}).md` | `Welcome-презентация (ФРИИ Спринт).md` |
| Исходные материалы | `{Тема} {Источник}/` (папка) | `Welcome-презентация ФРИИ/` |

### Алгоритм определения целевой базы знаний

```
1. Проанализировать содержимое материала
2. Задать вопросы:
   - Применим ли материал к ДРУГИМ проектам без изменений?
   - Содержит ли специфичный контекст (имена, даты, решения)?
   - Важно ли сохранить информацию об источнике?

3. Принять решение:
   - Если "да, да, да" → дублировать в обе БЗ (разные версии)
   - Если "да, нет, нет" → только общая БЗ
   - Если "нет, да, да" → только проектная БЗ
   
4. При дублировании:
   - Общая версия: универсальное имя, без контекста
   - Проектная версия: имя с источником, с контекстом и ссылками
```

## Примечания

- Все файлы должны быть импортированы в структуру ОС
- После импорта файлы удаляются из Инбокса или перемещаются в архив
- Имена файлов приводятся к единому формату согласно правилам именования
- Текстовые файлы могут быть конвертированы в Markdown для единообразия
- При импорте создаются необходимые директории автоматически
- **Интерактивность важнее скорости** — лучше задать лишний вопрос, чем сделать неправильный импорт
- **Разделение БЗ** — агент ДОЛЖЕН предлагать размещение, используя критерии выше, и получать подтверждение пользователя (кроме делегированного режима)
