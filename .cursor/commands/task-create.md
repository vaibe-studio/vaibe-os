# Алгоритм создания задачи

## Обзор
Команда `/task-create` создаёт задачу в AI операционной системе. Агент **интерактивно собирает информацию** о задаче, уточняет детали у пользователя и создаёт структурированную задачу с привязкой к проекту и роли.

**ВАЖНО**: Строго соблюдайте структуру файловой системы согласно правилам в `.cursor/rules/structure.md`.

---

## Формат вызова
- `/task-create {описание задачи}`
- `/task-create` (если описание не передано — агент обязан уточнить, что нужно сделать)

## Входы (источники данных)
- `Проекты/` — список проектов для выбора/валидации
- `Роли/` — список ролей для выбора/валидации
- `Проекты/{PROJECT}/README.md` — контекст проекта (если существует)
- `Роли/{ROLE}/README.md` и `Роли/{ROLE}/База знаний/` — контекст роли (если существует)
- `.cursor/rules/structure.md` — правила структуры и именования

## Выходы (артефакты)
- `Проекты/{PROJECT}/Задачи/{NUM}-{TITLE}/task.md` — созданная карточка задачи
- (опционально) создание нового проекта:
  - `Проекты/{PROJECT}/README.md` + базовая структура (`Задачи/`, `Встречи/`, `Исходные материалы/`, `База знаний/`)
- (опционально) создание новой роли:
  - `Роли/{ROLE}/README.md` + `Роли/{ROLE}/База знаний/`

---

## КРИТИЧЕСКИЕ ПРАВИЛА ИНТЕРАКТИВНОСТИ

> **ЗАПРЕЩЕНО**: Создавать задачу без подтверждения проекта пользователем.
> **ЗАПРЕЩЕНО**: Выбирать роль самостоятельно без согласования.
> **ЗАПРЕЩЕНО**: Создавать файлы до финального подтверждения плана.
> **ЗАПРЕЩЕНО**: Запрашивать подтверждение “создавать/не создавать” **до показа полного плана** (номер, путь, что будет создано, черновик содержания).
> **ЗАПРЕЩЕНО**: Пропускать этап уточнения, даже если контекст "очевиден".
> **ОБЯЗАТЕЛЬНО**: После каждого вопроса — ОСТАНОВИТЬСЯ и ДОЖДАТЬСЯ ответа.
> **ОБЯЗАТЕЛЬНО**: Использовать инструмент AskQuestion для структурированных вопросов.
> **ОБЯЗАТЕЛЬНО**: Показать финальный план и получить подтверждение перед созданием.

## Алгоритм

### 1. **Инициализация и анализ запроса**
   - Пользователь использует команду `/task-create` с описанием задачи
   - Агент получает контекст из workspace и открытых файлов
   - Агент определяет тип запроса: "Вопрос или Задача?"
     - **Если "Вопрос"** → ответ на основе базы знаний, завершение
     - **Если "Задача"** → продолжение алгоритма
   - **Критерии задачи:** глаголы действия (создать, разработать, исправить), измеримый результат

### 2. **Сбор контекста (ИНТЕРАКТИВНЫЙ)**

> ⚠️ **ТОЧКА ОСТАНОВА**: Агент ОБЯЗАН задать вопросы и ДОЖДАТЬСЯ ответа. НЕ продолжать без ответа пользователя.

   #### 2.1. Анализ и формирование гипотез
   - Сканирование существующих проектов в `Проекты/`
   - Сканирование существующих ролей в `Роли/`
   - Формулирование **гипотез** (но НЕ принятие решений):
     - Гипотеза о проекте
     - Гипотеза о роли
     - Гипотеза о приоритете

   #### 2.2. Запрос контекста у пользователя
   
   **Агент показывает анализ и задаёт вопросы (используя AskQuestion):**
   
   ```
   📋 Анализ задачи: {краткое описание понятой задачи}
   
   💡 Мои гипотезы:
   - Проект: {предположение или "новый проект"}
   - Роль: {предположение или "новая роль"}
   - Приоритет: {средний/высокий/низкий}
   
   Вопросы:
   1. Проект: {список существующих} или создать новый?
   2. Роль исполнителя: {список существующих} или создать новую?
   3. Приоритет: низкий / средний / высокий?
   4. Предлагаемое название задачи: "{моё предложение}" — согласны или предложите своё?
   5. Дополнительный контекст? (опционально)
   ```
   
   **→ СТОП. Ожидание ответа пользователя.**

### 3. **Определение проекта**
   **ТОЛЬКО ПОСЛЕ получения ответа** агент определяет проект:
   
   - **Если выбран существующий проект:**
     - Проверка существования `Проекты/{Название проекта}/`
     - Загрузка контекста из `README.md` проекта
   
   - **Если нужен новый проект:**
     
     > ⚠️ **ТОЧКА ОСТАНОВА**: Запросить детали нового проекта.
     
     ```
     🆕 Создание нового проекта
     
     Вопросы:
     1. Название проекта (на русском)?
     2. Краткое описание проекта?
     3. Основной контекст проекта?
     ```
     
     **→ СТОП. Ожидание ответа пользователя.**
     
     После получения ответа — план создания проекта (но НЕ создание):
     - Директория: `Проекты/{Название проекта}/`
     - Структура: `Задачи/`, `Встречи/`, `Исходные материалы/`, `База знаний/`
     - Индексный файл: `README.md`

### 4. **Определение роли**
   **ТОЛЬКО ПОСЛЕ определения проекта** агент уточняет роль:
   
   - **Если выбрана существующая роль:**
     - Проверка существования `Роли/{Название роли}/`
     - Загрузка контекста из `README.md` роли
   
   - **Если нужна новая роль:**
     
     > ⚠️ **ТОЧКА ОСТАНОВА**: Запросить детали новой роли.
     
     ```
     🆕 Создание новой роли
     
     Вопросы:
     1. Название роли (на русском)?
     2. Описание роли и зона ответственности?
     3. Ключевые навыки роли?
     ```
     
     **→ СТОП. Ожидание ответа пользователя.**

### 5. **Проверка базы знаний**
   - Поиск релевантных материалов в `Роли/{Название роли}/База знаний/`
   - Поиск в `База знаний/` (выборочно, если релевантно)
   - **Результат:** список найденных материалов для обогащения задачи

### 6. **Формирование плана задачи**
   Агент формирует полный план (но НЕ создаёт файлы):
   
   - Номер задачи: следующий номер в `Проекты/{NAME}/Задачи/`
   - Название задачи: согласованное с пользователем
   - Директория: `Проекты/{NAME}/Задачи/{NUM}-{TITLE}/`
   - Файл: `task.md` с обогащённым описанием

### 7. **Подтверждение перед созданием**

> ⚠️ **ТОЧКА ОСТАНОВА**: Перед созданием показать полный план и запросить подтверждение.

Правило формулировки:
- Вопрос подтверждения должен **включать** (в тексте формы AskQuestion или прямо перед ней) финальный план: проект/роль/приоритет/номер/путь и перечень создаваемых файлов.
- Если подтверждение было запрошено раньше по ошибке — **обязательно** повторить подтверждение после показа плана.

   **Агент показывает финальный план:**
   
   ```
   ✅ План создания задачи:
   
   📁 Проект: {название} (существующий/новый)
   👤 Роль: {название} (существующая/новая)
   🔢 Номер: {NUM}
   📝 Название: {TITLE}
   📂 Путь: Проекты/{NAME}/Задачи/{NUM}-{TITLE}/
   ⚡ Приоритет: {приоритет}
   
   📄 Содержимое task.md:
   - Описание: {краткое описание}
   - Требования: {список требований}
   - Контекст: {контекст из базы знаний}
   
   🆕 Будет создано:
   - [ ] Проект {название} (если новый)
   - [ ] Роль {название} (если новая)
   - [ ] Задача {NUM}-{TITLE}
   
   Выполнить создание? (да / нет / изменить)
   ```
   
   **→ СТОП. Ожидание подтверждения пользователя.**

### 8. **Создание задачи**
   **ТОЛЬКО ПОСЛЕ подтверждения** агент выполняет:
   
   - Создание проекта (если новый):
     - Директория `Проекты/{Название проекта}/`
     - Поддиректории: `Задачи/`, `Встречи/`, `Исходные материалы/`, `База знаний/`
     - Файл `README.md` с метаданными
   
   - Создание роли (если новая):
     - Директория `Роли/{Название роли}/`
     - Поддиректория `База знаний/`
     - Файл `README.md` с описанием
   
   - Создание задачи:
     - Директория `Проекты/{NAME}/Задачи/{NUM}-{TITLE}/`
     - Файл `task.md` со структурой:
       ```markdown
       # Задача: {Название}
       
       ## Метаданные задачи
       - **Номер**: {NUM}
       - **Проект**: {Название проекта}
       - **Роль**: {Название роли}
       - **Статус**: открыта
       - **Создана**: YYYY-MM-DD
       - **Приоритет**: {приоритет}
       
       ## Описание задачи
       {Детальное описание}
       
       ## Требования
       {Список требований}
       
       ## Контекст
       {Контекст из базы знаний и от пользователя}
       ```

### 9. **Завершение**
   - Подтверждение успешного создания
   - Показ пути к созданной задаче
   - Предложение: `/task-execute` для выполнения задачи
   - Если в проекте используется планирование через `Проекты/{PROJECT}/Планы/`:
     - предложить актуализировать snapshot-план: `/plan-update {PROJECT}`
     - пояснение: новые задачи, созданные через `/task-create`, могут отсутствовать в последнем плане до следующего `/plan-update`

## Режимы работы

В начале обработки агент может предложить режим (по контексту):

### Режим 1: Полный (по умолчанию)
- Все вопросы задаются последовательно
- Максимальный контроль пользователя
- Подходит для сложных задач и новых проектов

### Режим 2: Быстрый
- Агент предлагает план на основе гипотез
- Пользователь корректирует и подтверждает
- Подходит когда контекст очевиден (один проект, одна роль)
- **ВАЖНО**: Даже в быстром режиме ОБЯЗАТЕЛЬНО подтверждение перед созданием

## Примеры

### Пример 1: Задача в существующем проекте
**Запрос**: `/task-create Добавить валидацию форм на фронтенде`

1. Анализ: задача разработки, требует роль разработчика
2. Запрос контекста:
   ```
   📋 Анализ: Добавление валидации форм на фронтенде
   
   💡 Мои гипотезы:
   - Проект: ЕПК ФРИИ (единственный активный)
   - Роль: Разработчик
   - Приоритет: средний
   
   Вопросы:
   1. Проект: ЕПК ФРИИ — верно?
   2. Роль: Разработчик — верно?
   3. Название: "Валидация форм" — согласны?
   ```
3. Пользователь: "1. Да. 2. Да. 3. Лучше 'Клиентская валидация форм'"
4. Финальный план и подтверждение
5. Создание: `Проекты/ЕПК ФРИИ/Задачи/002-Клиентская валидация форм/task.md`

### Пример 2: Задача с новым проектом
**Запрос**: `/task-create Создать лендинг для нового продукта`

1. Анализ: задача дизайна и разработки, нужен новый проект
2. Запрос контекста:
   ```
   📋 Анализ: Создание лендинга для нового продукта
   
   💡 Мои гипотезы:
   - Проект: нужен новый (нет подходящего)
   - Роль: Веб-разработчик или Дизайнер
   
   Вопросы:
   1. Создать новый проект? Если да — как назвать?
   2. Какая роль будет исполнять: Веб-разработчик / Дизайнер / другая?
   ```
3. Пользователь: "1. Да, проект 'Лендинг АйТи'. 2. Создать роль 'Фуллстек разработчик'"
4. Уточнение деталей проекта и роли
5. Финальный план и подтверждение
6. Создание проекта, роли и задачи

### Пример 3: Быстрый режим
**Запрос**: `/task-create Исправить баг с авторизацией` (в контексте открыт файл проекта ЕПК ФРИИ)

1. Анализ: контекст очевиден — проект ЕПК ФРИИ, роль Разработчик
2. Предложение быстрого режима:
   ```
   🚀 Быстрый режим
   
   Проект: ЕПК ФРИИ
   Роль: Разработчик  
   Задача: 003-Исправление бага авторизации
   Приоритет: высокий (баг)
   
   Подтвердить создание? (да / нет / подробнее)
   ```
3. Пользователь: "да"
4. Создание задачи

## Обработка ошибок

### Если контекст неясен:
- Запрос дополнительной информации
- Предложение вариантов на основе анализа
- Возможность начать заново

### Если проект/роль не найдены:
- Предложение создать новый проект/роль
- Показ списка существующих для выбора

### Если пользователь отменяет:
- Корректное завершение без создания файлов
- Сохранение контекста для следующего запроса (опционально)

## АНТИПАТТЕРНЫ (чего НЕ делать)

### ❌ НЕПРАВИЛЬНО: Автоматический выбор проекта
```
Агент: "Вижу, что вы работаете над ЕПК ФРИИ. Создаю задачу..."
[сразу создаёт файлы]
```
**Проблема**: Пользователь не подтвердил проект, не выбрал название, не указал приоритет.

### ❌ НЕПРАВИЛЬНО: Создание без плана
```
Агент: "Задача создана! Путь: Проекты/ЕПК/Задачи/001-Задача/"
```
**Проблема**: Пользователь не видел план, не мог исправить название или детали.

### ❌ НЕПРАВИЛЬНО: Один общий вопрос
```
Агент: "Создать задачу в проекте ЕПК ФРИИ?"
Пользователь: "Да"
[агент сразу создаёт всё]
```
**Проблема**: Не уточнены роль, приоритет, название, требования.

### ✅ ПРАВИЛЬНО: Интерактивный цикл
```
Агент: "📋 Анализ задачи: Добавление валидации
       
       💡 Мои гипотезы:
       - Проект: ЕПК ФРИИ
       - Роль: Разработчик
       
       Вопросы:
       1. Проект: ЕПК ФРИИ — верно?
       2. Роль: Разработчик — верно?
       3. Название: 'Валидация форм' — согласны?"

Пользователь: "1. Да. 2. Да. 3. Лучше 'Клиентская валидация'"

Агент: "✅ План создания:
       - Путь: Проекты/ЕПК ФРИИ/Задачи/002-Клиентская валидация/
       - Роль: Разработчик
       
       Выполнить создание?"

Пользователь: "да"

Агент: [создаёт задачу]
```

    └── (пользовательские материалы)

## Связанные команды
- `/task-execute` — выполнить созданную задачу
- `/plan-update` — актуализировать snapshot-план проекта (чтобы новая задача попала в план/YouTrack поток)
- `/tasks-report` — посмотреть состояние задач проекта

---

## Примечания

- Все файлы создаются в формате Markdown
- **Все названия на русском языке**
- Номера задач с ведущими нулями (001, 002, 010)
- Даты в формате ISO 8601 (YYYY-MM-DD)
- **Интерактивность важнее скорости** — лучше уточнить, чем создать неправильно
