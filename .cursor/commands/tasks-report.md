# Отчёт по задачам проекта

## Обзор
Команда `/tasks-report` строит **детальный отчёт по задачам проекта** на основе карточек задач в `Проекты/{PROJECT}/Задачи/*/task.md` и факта наличия результатов (`results/`). Команда не создаёт “план недели”, а отвечает на вопросы:
- что открыто/в работе/на холде/выполнено
- где есть результаты и какие
- какие задачи требуют уточнения/следующего шага

## Формат вызова
- `/tasks-report`
- `/tasks-report {Название проекта}`

## Входы (источники данных)
- `Проекты/{PROJECT}/Задачи/*/task.md`
- `Проекты/{PROJECT}/Задачи/*/results/` (опционально)
- (опционально) `Проекты/{PROJECT}/README.md`
- (опционально) `Проекты/{PROJECT}/Планы/` — если нужно сверить с последним планом

## Выходы (артефакты)
- По умолчанию команда **не пишет файлы** — выводит отчёт в чат.
- (опционально) по просьбе пользователя может сохранить отчёт в:
  - `Проекты/{PROJECT}/Отчёты/Отчёт по задачам - {DD.MM.YYYY}.md`

## Критические правила
> **ЗАПРЕЩЕНО**: менять `task.md` без явной просьбы пользователя.
>
> **ОБЯЗАТЕЛЬНО**: если проект не указан — запросить выбор через `AskQuestion`.

## Алгоритм

### 1) Определение проекта
- Если проект не указан — показать список проектов из `Проекты/` и запросить выбор.

**→ СТОП.** Дождаться ответа.

### 2) Сканирование задач
- Найти все папки задач в `Проекты/{PROJECT}/Задачи/`
- Для каждой папки:
  - прочитать `task.md` (если нет — отметить как “повреждённая задача”)
  - определить статус (из метаданных/текста; если статус не найден — “не указан”)
  - проверить наличие `results/` и список файлов результатов

### 3) Нормализация статусов (для отчёта)
Использовать следующую нормализацию:
- **✅ выполнена**: явно указано в `task.md` ИЛИ есть `results/` с файлами и нет признаков “в работе”
- **🔄 в процессе**: явно указано в `task.md`
- **⏸️ на холде**: явно указано в `task.md`
- **⬜ открыта/не начата**: иначе
- **❓ требует уточнения**: явно указано в `task.md` или видно, что нет ключевых требований/входных данных

Если есть конфликт (например, `status: выполнена`, но нет результатов) — отметить как “⚠️ конфликт”.

### 4) Сбор сводных метрик
- количество задач по статусам
- процент выполненных
- топ-5 задач без следующего шага (если возможно определить)

### 5) Вывод отчёта
Вывести:
- краткое резюме
- таблицу задач по статусам (группами)
- список “⚠️ конфликты/проблемы данных”
- рекомендации “что сделать дальше”:
  - `/plan-update {PROJECT}` — если нужен управленческий план/приоритизация
  - `/task-execute {TASK}` — для выполнения выбранной задачи
  - `/yt-project-tasks-push` — если нужно опубликовать актуальные задачи (после актуализации плана)

## Примеры

```markdown
## Отчёт по задачам: {PROJECT}

### Резюме
- ✅ выполнено: 7
- 🔄 в процессе: 2
- ⏸️ на холде: 1
- ⬜ открыто: 12
- ⚠️ конфликты: 1

### 🔄 В процессе
| Задача | Путь | Результаты |
|-------|------|------------|
| 013-5 интервью с респондентами | Проекты/.../Задачи/013-.../task.md | results/: 2 файла |

### ⬜ Открыто
...

### ⚠️ Конфликты/проблемы
- 009-... помечена как “выполнена”, но папка results/ пуста.
```

## Обработка ошибок
- **Проект не найден**: показать список проектов и запросить выбор через `AskQuestion`.
- **Нет `Задачи/` или нет `task.md` в задаче**: пометить как “повреждённая структура” и не пытаться автоматически чинить без просьбы пользователя.
- **Конфликт статусов** (например, “выполнена”, но нет результатов): вывести в блок “⚠️ конфликты” и предложить варианты исправления (без автоматического редактирования).

## Связанные команды
- `/plan-update` — управленческий план проекта (приоритеты/спринт)
- `/task-execute` — выполнение конкретной задачи
- `/yt-project-tasks-push` и `/yt-project-tasks-pull` — синхронизация с YouTrack
