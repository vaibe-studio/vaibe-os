# Обновление плана проекта

## Обзор
Команда `/plan-update` создаёт или обновляет план проекта на основе актуального контекста.

## Формат вызова
- `/plan-update {Название проекта}`
- `/plan-update` (если проект не указан — выбор через `AskQuestion`)

## Входы (источники данных)
- `Проекты/{PROJECT}/README.md` — цели, команда, контекст
- `Проекты/{PROJECT}/Задачи/*/task.md` — канон карточек задач
- `Проекты/{PROJECT}/Задачи/*/results/` — факт результатов
- `Проекты/{PROJECT}/Встречи/*/tasks.md` и `summary.md` — задачи/договорённости со встреч
- `Проекты/{PROJECT}/Планы/` — предыдущие версии (если есть)
- `База знаний/Образец плана.md` — шаблон

## Выходы (артефакты)
- Новый snapshot-план в `Проекты/{PROJECT}/Планы/`:
  - `План - {N} - {DD.MM.YYYY} - {TITLE}.md`
- (опционально) секция “Изменения относительно версии N-1” в тексте плана

## Критические правила
> **ЗАПРЕЩЕНО**: редактировать старые планы “задним числом” — только создавать новую версию.
>
> **ОБЯЗАТЕЛЬНО**: подтягивать новые карточки задач из `Задачи/` в новый snapshot-план.
>
> **ОБЯЗАТЕЛЬНО**: если проект не указан — запросить выбор через `AskQuestion` (без автовыбора).

---

## Канон данных (важно для консистентности)

- **Каноническая карточка задачи**: `Проекты/{PROJECT}/Задачи/{NUM}-{TITLE}/task.md`
- **План**: `Проекты/{PROJECT}/Планы/План - N - ...md` — **управленческий snapshot** (приоритеты, текущий спринт, статусы) и точка интеграций (например, YouTrack).

Следствие:
- задачи могут существовать без попадания в план (например, только что созданные через `/task-create`);
- `/plan-update` должен **подтягивать фактические карточки задач** в новый snapshot-план;
- интеграции (`/yt-project-tasks-push`) должны работать с **последним актуальным планом**.

---

## Алгоритм

### 1. **Определение проекта**
   - Если проект указан — перейти к шагу 2
   - Если не указан — **обязательно запросить выбор проекта через форму**:
     - собрать список проектов из `Проекты/`
     - показать пользователю варианты (без догадок/автовыбора)
     - использовать `AskQuestion` (single-select), чтобы пользователь выбрал проект

### 2. **Проверка существующих планов**
   - Проверить папку `Проекты/{PROJECT}/Планы/`
   - Найти последний план по номеру версии в названии
   - Если планов нет — создать первый (N=1)
   - Если есть — создать следующую версию (N+1)

### 3. **Сбор данных для плана**
   
   #### 3.1 Метаданные проекта
   - `Проекты/{PROJECT}/README.md` — цели, команда, контекст
   
   #### 3.2 Формализованные задачи
   - `Проекты/{PROJECT}/Задачи/*/task.md` — все задачи проекта
   - `Проекты/{PROJECT}/Задачи/*/results/` — проверка наличия результатов
   
   #### 3.3 Задачи из встреч
   - `Проекты/{PROJECT}/Встречи/*/tasks.md` — задачи, назначенные на встречах
   - `Проекты/{PROJECT}/Встречи/*/summary.md` — ключевые решения и договорённости

   #### 3.4 Предыдущий план (если есть)
   - Загрузить последний план для сравнения статусов
   - Отметить прогресс по задачам

   #### 3.5 Проверка расхождений “карточки задач vs план”
   - Если существует предыдущий план:
     - сравнить список задач из папки `Задачи/` со списком задач, отражённых в плане
     - найти:
       - **новые задачи** (есть в `Задачи/`, но нет в плане)
       - **потерянные задачи** (есть в плане, но нет в `Задачи/` — возможна миграция/переименование/ошибка структуры)
   - Новые задачи обязаны попасть в новую версию плана (в подходящий раздел по приоритету/срочности).
   - Потерянные задачи помечаются в новой версии плана как “⚠️ требует проверки” (планы не удаляем и не ретконим).

### 4. **Анализ и классификация**
   
   #### 4.1 Определение статуса задач
   | Статус | Критерий |
   |--------|----------|
   | ✅ Выполнена | Есть папка `results/` с файлами |
   | ⏸️ На холде | Указано в task.md |
   | 🔄 В процессе | Есть частичные результаты |
   | ⬜ Не начата | Нет результатов |
   
   #### 4.2 Приоритизация
   | Приоритет | Критерии |
   |-----------|----------|
   | 🔴 Критический | Блокирует цель проекта, упоминается как срочное на встречах |
   | 🟠 Высокий | Важно для цели, срок — ближайшие дни |
   | 🟡 Средний | Поддерживает цель, срок — 1-2 недели |
   | 🟢 Низкий | Полезно, но не блокирует |
   
   #### 4.3 Оценка трудоёмкости
   - Анализ описания задачи
   - Сравнение с аналогичными выполненными задачами
   - Оценка в часах/днях
   
   #### 4.4 ИИ-возможность
   | Возможность | Критерии |
   |-------------|----------|
   | ✅ Полностью | Аналитика, исследования, генерация контента, документация |
   | 🔷 Частично | ИИ готовит черновик, человек дорабатывает/верифицирует |
   | ❌ Невозможно | Требуется личное взаимодействие, доступ к приватным данным |

### 5. **Генерация плана**
   
   - Использовать шаблон из `База знаний/Образец плана.md`
   - Заполнить все разделы актуальными данными:
     - I. Критические задачи на текущий спринт
     - II. Выполненные задачи
     - III. Срочные задачи (эта неделя)
     - IV. Среднесрочные задачи (1-2 недели)
     - V. Долгосрочные задачи (1 месяц+)
     - VI. Метрики успеха спринта
     - VII. Рекомендации по использованию ИИ
     - VIII. Приоритетный план на ближайшую неделю
     - IX. Важные напоминания

### 6. **Сохранение плана**
   
   **Формат имени файла:**
   ```
   План - {N} - {DD.MM.YYYY} - {TITLE}.md
   ```
   
   Где:
   - `{N}` — порядковый номер плана (1, 2, 3...)
   - `{DD.MM.YYYY}` — дата создания
   - `{TITLE}` — краткое название спринта/фокуса (3-5 слов)
   
   **Путь сохранения:**
   ```
   Проекты/{PROJECT}/Планы/
   ```

### 7. **Вывод результата**
   - Показать созданный план пользователю
   - Сообщить путь к сохранённому файлу
   - При обновлении — показать изменения относительно предыдущего плана

## Примеры

### Пример 1: Создание первого плана
**Запрос**: `/plan-update ЕПК ФРИИ`

1. Проект найден, планов нет → N=1
2. Сбор данных из README, Задачи/, Встречи/
3. Генерация плана по шаблону
4. Сохранение: `Планы/План - 1 - 27.01.2026 - Тактика продаж и интервью.md`

### Пример 2: Обновление плана
**Запрос**: `/plan-update ЕПК ФРИИ`

1. Проект найден, последний план: `План - 1 - 27.01.2026 - ...`
2. Сбор актуальных данных
3. Сравнение с предыдущим планом
4. Генерация нового плана с обновлёнными статусами
5. Сохранение: `Планы/План - 2 - 03.02.2026 - Интервью и эксперты.md`

### Пример 3: Без указания проекта
**Запрос**: `/plan-update`

1. Сформировать список проектов из `Проекты/`
2. Показать форму выбора (через `AskQuestion`, single-select)
3. После выбора — генерация/обновление плана

## Источники данных

### Обязательные:
- `Проекты/{PROJECT}/README.md`
- `Проекты/{PROJECT}/Задачи/*/task.md`
- `База знаний/Образец плана.md` — шаблон

### Опциональные (если есть):
- `Проекты/{PROJECT}/Задачи/*/results/`
- `Проекты/{PROJECT}/Встречи/*/tasks.md`
- `Проекты/{PROJECT}/Встречи/*/summary.md`
- `Проекты/{PROJECT}/Планы/` — предыдущие планы

---

## Рекомендации

1. **Регулярность**: Обновлять план в начале каждого спринта или после ключевых встреч
2. **Сравнение**: При обновлении сравнивать с предыдущим планом для отслеживания прогресса
3. **Фокус**: Использовать раздел VIII как план действий на неделю
4. **Делегирование**: Использовать раздел VII для распределения задач между ИИ и командой

---

## Связанные команды

- `/tasks-report` — детальный отчёт по задачам (без плана на неделю)
- `/task-create` — создание новой задачи
- `/task-execute` — выполнение задачи
- `/summarize-meeting` — создание саммари встречи с задачами
 - `/yt-project-tasks-push` — публикация задач в YouTrack (после актуализации плана)

См. `База знаний/Образец плана.md`

## Обработка ошибок
- **Нет проекта / проект не найден**: показать список проектов и запросить выбор.
- **Нет `Планы/`**: создать папку `Проекты/{PROJECT}/Планы/` и сформировать план N=1.
- **Нет задач**: создать план с пустыми разделами задач и явным блоком “Задачи отсутствуют/не заведены”.
- **Потерянные задачи (есть в плане, но нет в `Задачи/`)**: пометить в новой версии как “⚠️ требует проверки” и не удалять из истории.
