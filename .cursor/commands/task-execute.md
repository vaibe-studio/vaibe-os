# Алгоритм выполнения задачи

## Обзор
Команда `/task-execute` выполняет задачу в AI операционной системе. Агент **интерактивно согласовывает план**, выполняет задачу с использованием роли и базы знаний, **запрашивает подтверждение на критических этапах** и документирует результаты.

**ВАЖНО**: Строго соблюдайте структуру файловой системы согласно правилам в `.cursor/rules/structure.md`.

---

## Формат вызова
- `/task-execute {NUM}-{TITLE}`
- `/task-execute {NUM}`
- `/task-execute {часть названия}` (если однозначно находит задачу; иначе запросить выбор через AskQuestion)

## Входы (источники данных)
- `Проекты/{PROJECT}/Задачи/{NUM}-{TITLE}/task.md` — источник требований и статуса
- `Проекты/{PROJECT}/README.md` — контекст проекта (если есть)
- `Роли/{ROLE}/README.md` и `Роли/{ROLE}/База знаний/` — роль и практики (если применимо)
- `База знаний/` — выборочно, если релевантно

## Выходы (артефакты)
- Изменения в файлах проекта (код/документация) — **только после подтверждений**
- Обновление `task.md` (прогресс/результаты/статус) — только после подтверждения
- (опционально) файлы в `results/` внутри папки задачи

---

## КРИТИЧЕСКИЕ ПРАВИЛА ИНТЕРАКТИВНОСТИ

> **ЗАПРЕЩЕНО**: Начинать выполнение без согласования плана с пользователем.
> **ЗАПРЕЩЕНО**: Изменять файлы проекта без предварительного показа изменений.
> **ЗАПРЕЩЕНО**: Пропускать этап подтверждения, даже если план "очевиден".
> **ЗАПРЕЩЕНО**: Игнорировать блокирующие вопросы и продолжать выполнение.
> **ОБЯЗАТЕЛЬНО**: После каждого вопроса — ОСТАНОВИТЬСЯ и ДОЖДАТЬСЯ ответа.
> **ОБЯЗАТЕЛЬНО**: Использовать инструмент AskQuestion для структурированных вопросов.
> **ОБЯЗАТЕЛЬНО**: Показывать план и получать подтверждение перед выполнением.
> **ОБЯЗАТЕЛЬНО**: Запрашивать подтверждение перед изменением каждого файла (или группы файлов).

## Алгоритм

### 1. **Инициализация выполнения**
   - Пользователь использует команду `/task-execute` с номером/названием задачи
   - Агент загружает задачу из `Проекты/{NAME}/Задачи/{NUM}-{TITLE}/task.md`
   - Проверка статуса задачи:
     - **Выполнена** → информирование, предложение переоткрыть или выбрать другую
     - **В процессе** → загрузка текущего прогресса, продолжение
     - **Открыта** → начало выполнения

### 2. **Загрузка контекста**
   - Метаданные проекта: `Проекты/{NAME}/README.md`
   - Метаданные роли: `Роли/{Название роли}/README.md`
   - База знаний роли: `Роли/{Название роли}/База знаний/`
   - База знаний пользователя: `База знаний/` (выборочно, если релевантно)
   - Текущее состояние проекта (файлы, структура)

### 3. **Анализ и планирование (ИНТЕРАКТИВНЫЙ)**

> ⚠️ **ТОЧКА ОСТАНОВА**: Агент ОБЯЗАН показать план и ДОЖДАТЬСЯ подтверждения. НЕ начинать выполнение без ответа.

   #### 3.1. Анализ задачи
   - Разбор требований из `task.md`
   - Анализ зависимостей и ресурсов
   - Определение подзадач

   #### 3.2. Формирование плана
   
   **Агент показывает план выполнения:**
   
   ```
   📋 Задача: {название}
   📁 Проект: {проект}
   👤 Роль: {роль}
   
   📚 Загруженный контекст:
   - {список релевантных материалов из базы знаний}
   
   📝 План выполнения:
   1. {Подзадача 1} — {краткое описание}
   2. {Подзадача 2} — {краткое описание}
   3. {Подзадача 3} — {краткое описание}
   
   📂 Планируемые изменения:
   - Создание: {список новых файлов}
   - Изменение: {список изменяемых файлов}
   - Удаление: {список удаляемых файлов, если есть}
   
   ⏱️ Оценка сложности: {низкая/средняя/высокая}
   
   Вопросы:
   1. План выполнения корректен?
   2. Есть ли дополнительный контекст для выполнения?
   3. Режим: пошаговый (подтверждение каждого шага) / автономный (подтверждение в конце)?
   ```
   
   **→ СТОП. Ожидание ответа пользователя.**

### 4. **Применение роли и знаний**
   **ТОЛЬКО ПОСЛЕ подтверждения плана:**
   
   - Активация роли из `Роли/{Название роли}/README.md`
   - Загрузка применимых материалов из базы знаний:
     - Шаблоны и примеры
     - Лучшие практики
     - Архитектурные принципы
   - Адаптация знаний к контексту задачи

### 5. **Выполнение задачи (ИНТЕРАКТИВНЫЙ ЦИКЛ)**

> **СТОП-ПРАВИЛО**: В пошаговом режиме — подтверждение перед каждым изменением. В автономном режиме — подтверждение в конце перед записью.

   #### 5.1. Пошаговый режим (по умолчанию)
   
   Для каждой подзадачи:
   
   > ⚠️ **ТОЧКА ОСТАНОВА**: Показать изменения и запросить подтверждение.
   
   ```
   🔨 Подзадача {N}: {название}
   
   📄 Изменения:
   - Файл: {путь}
   - Действие: создание / изменение / удаление
   - Описание: {что будет сделано}
   
   [Предпросмотр изменений — если это код/текст]
   
   Выполнить? (да / нет / изменить / пропустить)
   ```
   
   **→ СТОП. Ожидание ответа.**
   
   После подтверждения:
   - Выполнение изменений
   - Обновление прогресса в `task.md`
   - Переход к следующей подзадаче

   #### 5.2. Автономный режим
   
   - Агент выполняет все подзадачи без промежуточных подтверждений
   - Собирает все изменения в итоговый отчёт
   - **НО**: перед записью файлов — обязательное подтверждение (см. шаг 6)

### 6. **Подтверждение изменений (для автономного режима)**

> ⚠️ **ТОЧКА ОСТАНОВА**: Перед записью файлов показать все изменения.

   ```
   ✅ Выполнение завершено
   
   📂 Итоговые изменения:
   
   1. Создано: {путь к файлу}
      [краткое описание содержимого]
   
   2. Изменено: {путь к файлу}
      [diff или описание изменений]
   
   3. Удалено: {путь к файлу}
      [причина удаления]
   
   Применить все изменения? (да / нет / показать подробнее / выборочно)
   ```
   
   **→ СТОП. Ожидание подтверждения.**

### 7. **Проверка качества**
   После применения изменений:
   - Проверка соответствия требованиям
   - Проверка кода (linting, форматирование) — `ReadLints`
   - Исправление найденных ошибок (с подтверждением)
   - Проверка полноты реализации

## Частый кейс: результат нужен в PDF

Если пользователь просит артефакт “в формате PDF”:

1) Сформировать результат **в Markdown** в `results/` задачи (исходник остаётся в репозитории).
2) Сгенерировать PDF через встроенную утилиту репозитория:

```powershell
python -m tools.markdown_to_pdf "input.md" -o "output.pdf"
```

Примечания:
- Этот путь не требует `pandoc` и корректно работает с кириллицей (Windows-шрифты).
- Перед запуском при необходимости: `python -m pip install -r tools/markdown_to_pdf/requirements.txt`
- Для всех shell-команд на Windows учитывать `.cursor/rules/powershell.md` (в т.ч. про кириллические пути).

### 8. **Документирование результатов**

> ⚠️ **ТОЧКА ОСТАНОВА**: Показать итоговый отчёт перед записью в task.md.

   **Агент показывает результаты:**
   
   ```
   📊 Результаты выполнения
   
   ✅ Выполнено:
   - {подзадача 1} — выполнено
   - {подзадача 2} — выполнено
   - {подзадача 3} — выполнено
   
   📂 Созданные/изменённые файлы:
   - {список файлов}
   
   ⚠️ Проблемы и решения:
   - {проблема}: {решение}
   
   📝 Обновить task.md с этими результатами?
   ```
   
   **→ СТОП. Ожидание подтверждения.**
   
   После подтверждения — обновление `task.md`:
   ```markdown
   ## Результаты выполнения
   - [Что было сделано]
   - [Созданные файлы]
   - [Изменения]
   
   ## Проблемы и решения
   - [Проблема 1]: [Решение]
   
   ## Статус
   - **Статус**: выполнена
   - **Завершена**: YYYY-MM-DD
   ```

### 9. **Обновление базы знаний**
   Если в процессе найдены полезные паттерны:
   
   ```
   💡 Предложение обновить базу знаний
   
   Найденные паттерны/решения:
   - {паттерн 1}: {описание}
   
   Добавить в Роли/{Роль}/База знаний/? (да / нет)
   ```
   
   **→ СТОП. Ожидание ответа.**

### 10. **Завершение выполнения**
   - Обновление статуса задачи на "выполнена"
   - Показ итогового статуса
   - Предложение следующих шагов:
     - Связанные задачи
     - `/task-create` для новой задачи
     - `/inbox-check` если есть материалы для импорта

## Режимы выполнения

В начале выполнения агент ОБЯЗАН предложить режим:

### Режим 1: Пошаговый (по умолчанию)
- Подтверждение перед каждым изменением файла
- Максимальный контроль пользователя
- Подходит для критических изменений и обучения

### Режим 2: Автономный
- Агент выполняет все подзадачи самостоятельно
- Одно итоговое подтверждение перед записью всех файлов
- Подходит для рутинных и понятных задач
- **ВАЖНО**: Даже в автономном режиме ОБЯЗАТЕЛЬНО показать изменения перед записью

### Режим 3: Доверительный (только для опытных)
- Агент выполняет и записывает изменения без промежуточных подтверждений
- Итоговый отчёт после выполнения
- Пользователь явно говорит "выполни без подтверждений"
- **ВАЖНО**: Агент должен предупредить о рисках

## Примеры

### Пример 1: Пошаговый режим
**Запрос**: `/task-execute 002-Клиентская валидация форм`

1. Загрузка задачи и контекста
2. Показ плана:
   ```
   📋 Задача: Клиентская валидация форм
   📁 Проект: ЕПК ФРИИ
   👤 Роль: Разработчик
   
   📝 План:
   1. Создать утилиты валидации
   2. Добавить валидацию в форму регистрации
   3. Добавить валидацию в форму авторизации
   
   Режим: пошаговый / автономный?
   ```
3. Пользователь: "пошаговый"
4. Подзадача 1:
   ```
   🔨 Подзадача 1: Создать утилиты валидации
   
   📄 Создание: src/utils/validation.ts
   [предпросмотр кода]
   
   Выполнить?
   ```
5. Пользователь: "да"
6. Выполнение → следующая подзадача → ... → завершение

### Пример 2: Автономный режим
**Запрос**: `/task-execute 003-Исправление бага авторизации`

1. Загрузка задачи
2. Показ плана, пользователь выбирает "автономный"
3. Агент выполняет все подзадачи
4. Итоговое подтверждение:
   ```
   ✅ Выполнение завершено
   
   📂 Изменения:
   1. Изменено: src/auth/login.ts
      - Исправлена проверка токена
   2. Изменено: src/auth/session.ts
      - Добавлен refresh токена
   
   Применить? (да / нет / показать diff)
   ```
5. Пользователь: "да"
6. Запись файлов, документирование

### Пример 3: Задача требует уточнения
**Запрос**: `/task-execute 001-Интеграция платежной системы`

1. Загрузка задачи
2. Агент обнаруживает неполноту требований:
   ```
   ⚠️ Требуется уточнение
   
   Задача: Интеграция платежной системы
   
   Недостающая информация:
   1. Какую платёжную систему использовать? (Stripe / YooKassa / другая)
   2. Какие способы оплаты поддерживать?
   3. Есть ли тестовые ключи API?
   
   Пожалуйста, уточните детали.
   ```
3. **→ СТОП. Ожидание ответа.**

## Обработка ошибок

### Если задача не найдена:
- Показать список доступных задач в проекте
- Предложить `/task-create` для создания новой

### Если задача не может быть выполнена:
- Фиксация причины невозможности
- Предложение альтернативных решений
- Обновление статуса на "требует уточнения"
- Запрос информации у пользователя

### Если требуется помощь пользователя:
- Чёткое описание блокера
- Предложение вариантов решения
- **→ СТОП. Ожидание ответа перед продолжением.**

### Если возникла ошибка при выполнении:
- Откат изменений (если возможно)
- Показ ошибки пользователю
- Предложение исправления

## АНТИПАТТЕРНЫ (чего НЕ делать)

### ❌ НЕПРАВИЛЬНО: Выполнение без плана
```
Агент: "Начинаю выполнение задачи..."
[сразу изменяет файлы]
```
**Проблема**: Пользователь не видел план, не мог скорректировать подход.

### ❌ НЕПРАВИЛЬНО: Изменение файлов без подтверждения
```
Агент: "Задача выполнена! Изменены файлы: auth.ts, login.ts"
```
**Проблема**: Пользователь не видел изменения до их применения.

### ❌ НЕПРАВИЛЬНО: Игнорирование блокеров
```
Агент: "Не нашёл API ключ, но продолжаю выполнение..."
[создаёт код с заглушками]
```
**Проблема**: Нужно было остановиться и запросить ключ.

### ✅ ПРАВИЛЬНО: Интерактивный цикл
```
Агент: "📋 План выполнения:
       1. Создать утилиты валидации
       2. Добавить в форму регистрации
       
       Режим: пошаговый / автономный?"

Пользователь: "пошаговый"

Агент: "🔨 Подзадача 1: Создать утилиты
       
       📄 Создание: src/utils/validation.ts
       [код]
       
       Выполнить?"

Пользователь: "да"

Агент: [создаёт файл, переходит к следующей подзадаче]
```

## Отслеживание прогресса

Во время выполнения агент ОБЯЗАН:
- Обновлять статус подзадач в `task.md` после каждого шага
- Информировать пользователя о прогрессе
- Фиксировать важные решения
- Сохранять промежуточные результаты

Формат прогресса в `task.md`:
```markdown
## План выполнения
1. [Подзадача 1] ✓ — выполнено
2. [Подзадача 2] → в процессе
3. [Подзадача 3] ⏳ — ожидает
```

---

## Структура обновления задачи

```markdown
# Задача: [Название]

## Метаданные задачи
- **Номер**: {NUM}
- **Проект**: {Название проекта}
- **Роль**: {Название роли}
- **Статус**: в процессе / выполнена / требует уточнения
- **Создана**: YYYY-MM-DD
- **Начата**: YYYY-MM-DD
- **Завершена**: YYYY-MM-DD (если выполнена)
- **Приоритет**: средний/высокий/низкий

## Описание задачи
[Описание]

## План выполнения
1. [Подзадача 1] ✓
2. [Подзадача 2] ✓
3. [Подзадача 3] ✓

## Результаты выполнения
[Результаты]

## Проблемы и решения
[Проблемы и их решения]

## Связанные файлы
- [Список созданных/измененных файлов]
```

## Связанные команды
- `/task-create` — создать новую задачу
- `/plan-update` — обновить snapshot-план проекта (приоритеты/спринт)
- `/inbox-check` — импортировать материалы в структуру (если задача принесла новые файлы)
- `/evolve` — зафиксировать найденный паттерн/улучшение методологии

---

## Примечания

- Агент должен быть прозрачным в своих действиях
- Все изменения должны быть документированы
- Промежуточные результаты должны сохраняться
- При необходимости можно приостановить и возобновить позже
- **Интерактивность важнее скорости** — лучше показать план, чем сделать неправильно
