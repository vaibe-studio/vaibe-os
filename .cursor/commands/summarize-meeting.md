# Саммаризация встреч

## Обзор
Команда `/summarize-meeting` находит **несаммаризированные встречи** в проектах и для каждой встречи формирует:
- `summary.md` — итоги, решения, риски, следующие шаги
- `tasks.md` — список задач по итогам встречи (с привязкой к владельцам/срокам)

Команда **работает интерактивно**: агент предлагает, пользователь подтверждает/правит перед записью файлов.

## Формат вызова
- `/summarize-meeting`
- `/summarize-meeting {Название проекта}` (опционально)

## Входы (источники данных)
- `Проекты/*/Встречи/*/transcript.md`
- `Проекты/*/Встречи/*/transcript.txt`
- (опционально) `Проекты/*/README.md` — контекст проекта

## Выходы (артефакты)
- `Проекты/{PROJECT}/Встречи/{MEETING}/summary.md`
- `Проекты/{PROJECT}/Встречи/{MEETING}/tasks.md`

## Критические правила (безопасность и интерактивность)
> **ЗАПРЕЩЕНО**: писать `summary.md`/`tasks.md` без подтверждения пользователя.
>
> **ЗАПРЕЩЕНО**: “додумывать” владельцев и сроки без явной пометки гипотезы и подтверждения.
>
> **ОБЯЗАТЕЛЬНО**: использовать `AskQuestion` для выбора проекта/встреч/режима.
>
> **ТОЧКА ОСТАНОВА**: перед записью каждого `summary.md` и `tasks.md` — показать черновик и дождаться подтверждения.

## Определение “несаммаризированной встречи”
Встреча считается несаммаризированной, если выполняется любое из условий:
- отсутствует `summary.md` **или** `tasks.md`
- файл существует, но пустой
- файл содержит только placeholder (например, “TBD”, “TODO”, “заполнить”)

## Алгоритм

### 1) Выбор проекта (или всех проектов)
- Если проект передан параметром — работаем с ним.
- Если не передан — показать список проектов из `Проекты/` и спросить, что обработать:
  - один проект
  - несколько проектов
  - все проекты

**→ СТОП.** Дождаться ответа.

### 2) Поиск встреч и кандидатов на саммари
Для выбранных проектов:
- просканировать `Проекты/{PROJECT}/Встречи/*/`
- найти папки встреч, где есть `transcript.md` или `transcript.txt`
- определить, какие встречи “несаммаризированы” по правилам выше

Показать список кандидатов (таблицей) и спросить, какие обрабатывать:
- все
- выборочно
- отмена

**→ СТОП.** Дождаться ответа.

### 3) Режим выполнения
Запросить режим:
- **Пошаговый (по умолчанию)**: подтверждение для каждой встречи отдельно
- **Пакетный**: агент готовит черновики по всем выбранным встречам и затем одно подтверждение на запись файлов

**→ СТОП.** Дождаться ответа.

### 4) Обработка встречи (цикл)
Для каждой выбранной встречи:

#### 4.1 Извлечение контекста
- определить метаданные встречи из имени папки (дата, номер/тема — если есть)
- прочитать transcript
- выделить:
  - цель/повод звонка
  - ключевые тезисы
  - решения/договорённости
  - вопросы и открытые пункты
  - явные action items (задачи)

#### 4.2 Уточнение у пользователя (минимум)
Показать краткую выжимку и задать вопросы (через `AskQuestion` или текстом):
- тема встречи сформулирована корректно?
- нужно ли добавить/исправить участников (опционально)?
- подтверждаем список задач? что исправить: владельцы/сроки/формулировки/приоритет?

**→ СТОП.** Дождаться ответа.

#### 4.3 Черновики файлов
Сформировать черновики `summary.md` и `tasks.md` по шаблонам ниже.

#### 4.4 Подтверждение перед записью
Показать:
- путь файлов
- черновик `summary.md`
- черновик `tasks.md`

Спросить:
- записать как есть
- правки
- пропустить встречу

**→ СТОП.** Дождаться ответа.

#### 4.5 Запись результатов
После подтверждения:
- создать/обновить `summary.md`
- создать/обновить `tasks.md`
- (опционально) предложить создать задачи в `Проекты/{PROJECT}/Задачи/` через `/task-create` для 1–3 самых важных action items

### 5) Итоговый отчёт
Показать сводку:
- какие встречи обработаны
- где лежат файлы
- какие задачи выделены (с количеством)
- что осталось несаммаризированным

## Шаблон `summary.md`

```markdown
# Саммари встречи: {ТЕМА}

## Контекст
- **Проект**: {PROJECT}
- **Дата**: {DD.MM.YYYY} (если известна)
- **Источник**: transcript.{md|txt}
- **Участники**: {если известны; иначе “не указано”}

## Итоги (кратко)
- ...

## Решения и договорённости
- ...

## Риски / блокеры
- ...

## Следующие шаги
- ...

## Открытые вопросы
- ...
```

## Шаблон `tasks.md`

```markdown
# Задачи по итогам встречи: {ТЕМА}

## Список задач

| № | Задача | Владелец | Срок | Приоритет | Статус | Примечание |
|---|--------|----------|------|-----------|--------|------------|
| 1 | ... | ... | ... | 🔴/🟠/🟡/🟢 | ⬜ | ... |

## Привязка к vAIbe-os (опционально)
- Если нужно превратить задачу в полноценную карточку: используйте `/task-create` и вставьте ссылку на `Проекты/{PROJECT}/Задачи/{NUM}-{TITLE}/task.md`.
```

## Обработка ошибок
- **Нет transcript**: сообщить и пропустить встречу (не создавать пустые summary/tasks).
- **Transcript слишком большой/шумный**: предложить пакетный режим + саммари “в 2 прохода” (сначала грубо, затем уточнение).
- **Неясные задачи**: пометить как гипотезу в `tasks.md` и запросить подтверждение у пользователя.

## Примеры
- **Один проект**: `/summarize-meeting ЕПК ФРИИ` → найти кандидатов → пошагово подтвердить `summary.md` и `tasks.md`.
- **Все проекты**: `/summarize-meeting` → выбрать “все проекты” → пакетный режим → одно подтверждение на запись.

## Связанные команды
- `/plan-update` — обновить план проекта с учётом новых задач/результатов
- `/task-create` — создать карточку задачи из пункта `tasks.md`
- `/yt-project-tasks-push` — опубликовать задачи (если план/синк настроены)
