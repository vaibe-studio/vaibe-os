# Участие в vAIbe-OS

> :gb: [Read in English](CONTRIBUTING.md)

Спасибо за интерес к vAIbe-OS! Проект построен на философии **партнёрства человека и ИИ** — это распространяется и на работу с контрибьюторами.

## Золотое правило: правьте канон, а не обёртки

vAIbe-OS работает по модели **один канон, много нативных обёрток**. Единственный источник правды — `.vaibe/` (правила, скиллы, агенты, скрипты). Слои под IDE — `.claude/`, `.cursor/`, `.codex/`, `.opencode/` и шимы `CLAUDE.md` — **генерируются** и помечены маркером `GENERATED`. Не правьте их руками: изменения сотрутся при следующей перегенерации.

После любого изменения канона перегенерируйте и проверьте встроенным **doctor**:

```bash
D=.vaibe/scripts/doctor
uv run --project $D $D/main.py treat       # перегенерировать обёртки из канона
uv run --project $D $D/main.py diagnose    # проверить целостность канон ↔ нативка (должно быть зелено)
```

`diagnose` — это CI-гейт: PR обязан держать его зелёным. Коммитьте канон **и** перегенерированные обёртки вместе.

## Как внести вклад

### 1. Сообщить о баге или предложить улучшение

[Создайте issue](../../issues/new/choose) — есть шаблоны для багов, фич и новых скиллов.

### 2. Добавить новый скилл

Скиллы — сердце vAIbe-OS. Каждый скилл — это папка в `.vaibe/skills/` с плейбуком `SKILL.md`.

**Шаги:**
1. Создайте `.vaibe/skills/your-skill/SKILL.md` (имя папки совпадает с полем `name`, латиница в kebab-case)
2. Добавьте YAML-метаданные:
   ```yaml
   ---
   name: your-skill
   description: Что делает скилл + ключевые слова. Triggers: keyword1, keyword2.
   license: MIT
   ---
   ```
3. Включите секции: **Purpose**, **Procedure** (пошагово), **Output format**, **Quality bar**
4. Вспомогательные скрипты — в `.vaibe/skills/your-skill/scripts/`, справочные материалы — в `references/` или `assets/` (на каждый такой файл нужна входящая ссылка)
5. Перегенерируйте нативный слой (`doctor treat`) и убедитесь, что `doctor diagnose` зелёный
6. Отправьте PR

Агент находит скиллы сам по их `description` — ни роутера, ни реестра обновлять не нужно.

**Примеры хороших скиллов:** смотрите существующие в `.vaibe/skills/`.

### 3. Улучшить существующие скиллы или знания

Все скиллы — Markdown-файлы в `.vaibe/skills/`. Справочные/доменные скиллы (стратегия, проектное управление, продажи и др.) лежат рядом с прикладными. Прочитайте, улучшите, перегенерируйте, отправьте PR.

### 4. Улучшить скрипты

Python-утилиты находятся в `.vaibe/scripts/{name}/` (кросс-скилловые) или `.vaibe/skills/{skill}/scripts/`. Каждый — **самодостаточный `uv`-проект** со своим `pyproject.toml`; запуск:

```bash
uv run --project .vaibe/scripts/<name> .vaibe/scripts/<name>/main.py [args]
```

- Следуйте стилю существующего кода; держите зависимости минимальными
- Коммитьте `uv.lock`
- Никаких `requirements.txt`, ручных venv и `python -m tools.X` (PEP 723 inline-метаданные — только для одноразовых однострочников)

### 5. Улучшить документацию

README, AGENTS.md, файлы правил в `.vaibe/rules/` — улучшения ясности и точности всегда приветствуются.

## Правила

### Соответствие философии

vAIbe-OS построен на конкретных принципах ([Манифест](.vaibe/rules/manifesto.md), [Онтология](.vaibe/rules/ontology.md)):

- **Партнёрство, не замена** — фичи должны усиливать пользователя, не создавать зависимость
- **Аддитивная эволюция** — добавляем и уточняем, а не удаляем. Знания накапливаются
- **Автономия источника** — решение всегда за пользователем. Не создавайте фичи, которые решают за него
- **Честность об LLM** — признавайте ограничения, не создавайте иллюзий

### Именование файлов

- Папки и файлы в хранилище — **на русском языке** (кроме канона `.vaibe/`, `repositories/` и системных `.md` — они на латинице)
- Номера задач с ведущими нулями: `001`, `002`, `010`
- Без переименований, отличающихся только регистром (контрибьюторы есть и на Windows, и на Linux)
- Полная спецификация: [`.vaibe/rules/structure.md`](.vaibe/rules/structure.md)

### Pull requests

- Один PR — одно изменение
- Опишите **что** и **зачем** в описании PR
- При изменении скиллов или правил — объясните, как это соответствует манифесту
- Для новых скиллов: приложите хотя бы один пример использования
- Приложите перегенерированный нативный слой и держите `doctor diagnose` зелёным
- **Никогда `git add -A`** — добавляйте явные пути (в хранилище есть личные/игнорируемые зоны)

### Стиль кода

- Python: следуйте существующим паттернам, держите зависимости минимальными
- Markdown: ATX-заголовки (`#`), огороженные блоки кода
- Коммиты: [Conventional Commits](https://www.conventionalcommits.org/) с эмодзи-префиксом (см. [`.vaibe/rules/git-commits.md`](.vaibe/rules/git-commits.md))

## Настройка окружения

```bash
git clone https://github.com/vaibe-studio/vaibe-os.git
cd vaibe-os
```

Для запуска скриптов и doctor нужен [`uv`](https://docs.astral.sh/uv/) (он сам управляет окружением каждого скрипта). Само ядро хранилища — просто Markdown, сборка не нужна.
