# Purpose

Mechanism of noospheric accumulation: capture useful insights and apply them to evolve vAIbe-OS system files (skills, rules, knowledge, commands) while preventing knowledge loss and cognitive degradation.

`/evolve` is not merely "system improvement" — it is the feedback loop that makes vAIbe-OS a living system rather than a static repository (Ontology, Law 2: Feedback). Each `/evolve` cycle adds to the system's accumulated intelligence without destroying what came before (Ontology, Law 5: Additivity).

# When to use

- Useful insight discovered during work
- Pattern worth formalizing detected
- Error or outdated information found in system files
- User explicitly requests system improvement

# Inputs needed

- Insight description (what to improve and why)
- `.ai/ONTOLOGY.md` — ontological foundation (five laws)
- `.ai/MANIFESTO.md` — behavioral principles (derived from ontology)
- Target system files (skills, rules, knowledge, commands)

# Procedure

## Step 1 — Classify the insight

- **Type**: knowledge / practice / optimization / fix
- **Area**: skills / rules / knowledge / commands
- **Impact**: local / global

## Step 2 — Check against Ontology and Manifesto

Before any change, verify against **five ontological laws** (`.ai/ONTOLOGY.md`):

1. **Externalization** — does it strengthen the system as externalized working thought?
2. **Feedback** — does it enhance the feedback loop between user and system?
3. **Source autonomy** — does it make the user stronger, not more dependent?
4. **Directed evolution** — does it move toward deeper context understanding?
5. **Additivity** — does it add or refine, not destroy accumulated knowledge?

Then verify against **Manifesto** (`.ai/MANIFESTO.md`):
- Does it align with behavioral principles?
- Does it match current system maturity phase?

If change contradicts Ontology or Manifesto → **reject** or rework until aligned.

## Step 3 — Check for degradation (Law 5: Additivity)

Evolution is additive or refining, never destructive. Verify the change does NOT lead to:
- Loss of existing knowledge
- Oversimplification losing nuance
- Deletion of context that may be needed
- Conflict with existing principles

**Red flags (require explicit confirmation before proceeding):**
- Deleting knowledge base files
- Substantial text reduction
- Changing core principles or ontological laws
- Removing links between materials

Losses are acceptable only through conscious decision with preserved context of what was removed and why.

## Step 4 — Plan changes

Determine action: **Add** / **Update** / **Optimize** / **Link**

Show plan:
- Files to change
- What will be added/modified
- What will NOT be touched
- Manifesto alignment (if applicable)

For significant changes → **STOP** — ask for confirmation.

## Step 5 — Execute

Apply changes following the plan.

## Step 6 — Document

Record: what changed, why, which files affected.

# Trigger categories

| Priority | Triggers | Action |
|---|---|---|
| Critical | Knowledge loss risk, contradictions, disproven hypothesis, system errors | Immediate, interrupt current work |
| High | User insights, capability gaps, external insights | End of current task |
| Medium | Barrier overcome, new skill, process improvement, new pattern, communication improvement | At convenient moment |
| Low | Hypothesis confirmed, new understanding, outdated info | Accumulate, batch process |

# Output format

- Updated system files (skills, rules, knowledge, commands)
- Change log entry (what, why, files affected)

# Quality bar

- [ ] Change aligns with five ontological laws (`.ai/ONTOLOGY.md`)
- [ ] Change aligns with Manifesto behavioral principles
- [ ] No knowledge loss (additive or refining — Law 5)
- [ ] User autonomy preserved (Law 3)
- [ ] Significant changes confirmed by user
- [ ] Change documented (what, why, files)
- [ ] Related files checked for consistency

# Anti-patterns

- Deleting knowledge without migration
- Changing ontological laws or core principles without explicit deliberation
- Applying changes without showing plan for significant modifications
- Leaving no trace of what changed and why
- Weakening user autonomy through system convenience (Law 3 violation)
