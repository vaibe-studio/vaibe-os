# PowerShell: совместимость команд на Windows

Цель: избежать повторяющихся ошибок при выполнении shell-команд в среде PowerShell (Windows).

## Определение среды

Если в `user_info` указано `Shell: powershell` или `OS Version: win32` — все команды выполняются в PowerShell. Применяй правила ниже.

## Запрещённые конструкции

| Конструкция | Почему ломается | Что использовать |
|-------------|----------------|-----------------|
| `<<'EOF' ... EOF` (heredoc) | PowerShell не поддерживает bash heredocs | Записать текст в файл → использовать файл |
| `&&` (цепочка команд) | Не работает в PowerShell < 7 | Выполнять команды **отдельными вызовами Shell** |
| `-m "длинный русский текст с (скобками)"` | PS интерпретирует `(...)` как subexpression, кириллица может ломать парсер | Использовать `git commit -F <файл>` |
| `$(command)` внутри строк | PS подставляет свои переменные / выражения | Избегать; использовать промежуточные файлы |

## Git commit с русским текстом (рекомендуемый паттерн)

### Шаг 1: записать сообщение в файл

Использовать инструмент **Write** (не echo/heredoc) для создания временного файла:

```
Путь: .commit_msg.tmp
Содержимое: текст коммит-сообщения (любой длины, любые символы)
```

### Шаг 2: staging (отдельный вызов Shell)

```powershell
git add "путь/к/файлам/"
```

### Шаг 3: commit (отдельный вызов Shell)

```powershell
git commit -F .commit_msg.tmp
```

### Шаг 4: проверка + очистка (отдельный вызов Shell)

```powershell
git status
```

После успешного коммита удалить `.commit_msg.tmp` через инструмент **Delete**.

## Кириллические пути: файловые операции (КРИТИЧЕСКИ ВАЖНО)

**Корневая причина**: Shell tool сохраняет команды во временный `.ps1`-файл. Кириллические символы в путях искажаются при записи (double-encoding / Mojibake). Это затрагивает **все** shell-команды: `copy`, `Copy-Item`, `move`, `Move-Item`, `python shutil.copy(прямой_текст)`, `cmd /c copy` и т.д.

### Что работает и что НЕ работает

| Операция | Кириллица прямым текстом | Unicode-эскейпы в Python |
|----------|:------------------------:|:------------------------:|
| `Copy-Item` / `copy` | ❌ | — |
| `Move-Item` / `move` | ❌ | — |
| `cmd /c copy` | ❌ | — |
| `python shutil.copy2()` | ❌ | ✅ `\uXXXX` |
| `python shutil.move()` | ❌ | ✅ `\uXXXX` |
| `python os.makedirs()` | ❌ | ✅ `\uXXXX` |
| `mkdir` (PowerShell) | ❌ (может создать мусорные папки) | — |
| `git add` | ❌ (часто) | ✅ через `subprocess` |
| Cursor Read / Write / Delete | ✅ (нативно) | — |
| Cursor Glob | ⚠️ ненадёжен (см. ниже) | — |
| Cursor Grep | ⚠️ ненадёжен (см. ниже) | — |

### Cursor Glob / Grep и кириллические пути (КРИТИЧЕСКИ ВАЖНО)

Инструменты **Glob** и **Grep** Cursor могут **не находить файлы** с кириллическими именами на Windows. Это баг индексатора Cursor — не зависит от настроек терминала или кодовых страниц.

**Симптомы:**
- `Glob("Проекты/vAIbe-OS/Задачи/018-*/**")` → 0 results, хотя файлы существуют
- Часть файлов находится (например, 001–017), а другая часть — нет (018–022)
- Закономерность непредсказуема: может зависеть от времени индексации, длины пути, конкретных символов

**Правила:**
1. **НИКОГДА** не делать вывод "файл/папка не существует" только на основании Glob/Grep с нулевым результатом
2. При 0 results для кириллического пути — **обязательно верифицировать** через PowerShell:
   ```powershell
   [Console]::OutputEncoding = [System.Text.Encoding]::UTF8
   Get-ChildItem -Name "Проекты\vAIbe-OS\Задачи\"
   ```
3. Для листинга директорий с кириллическими именами предпочитать PowerShell с `[Console]::OutputEncoding = UTF8` или Python `os.listdir()` с Unicode-эскейпами
4. Glob **также** уважает `.gitignore` — директории с `*` в `.gitignore` (например, `Инбокс/`) будут пустыми для Glob

### Выбор способа по типу операции

#### Копирование / перемещение файлов → Python shutil (ПРИОРИТЕТ 1)

> **КРИТИЧНО**: НЕ использовать Read + Write для копирования/перемещения файлов!
> Read + Write заставляет AI перегонять весь контент через себя — для файла в 1000+ строк это минуты ожидания, расход токенов и риск случайных изменений.

**Копирование файла** (1 секунда, файл копируется напрямую на диске):

```python
python -c "import shutil, os; shutil.copy2(os.path.join('c:\\vaibe-os', '\u0418\u043d\u0431\u043e\u043a\u0441', 'file.md'), os.path.join('c:\\vaibe-os', '\u041f\u0440\u043e\u0435\u043a\u0442\u044b', 'МойПроект', '\u0412\u0441\u0442\u0440\u0435\u0447\u0438', 'dest.md'))"
```

**Перемещение файла** (1 секунда):

```python
python -c "import shutil, os; shutil.move(os.path.join('c:\\vaibe-os', '\u0418\u043d\u0431\u043e\u043a\u0441', 'file.md'), os.path.join('c:\\vaibe-os', '\u041f\u0440\u043e\u0435\u043a\u0442\u044b', 'dest.md'))"
```

**Создание директории**:

```python
python -c "import os; os.makedirs(os.path.join('c:\\vaibe-os', '\u041f\u0440\u043e\u0435\u043a\u0442\u044b', 'МойПроект', '\u0412\u0441\u0442\u0440\u0435\u0447\u0438', 'New Folder'), exist_ok=True)"
```

**Git add**:

```python
python -c "import subprocess; subprocess.run(['git', 'add', '--', '\u041f\u0440\u043e\u0435\u043a\u0442\u044b/МойПроект/README.md'])"
```

#### Создание новых файлов, редактирование, чтение содержимого → Cursor-инструменты

Cursor-инструменты работают нативно с кириллицей и идеальны когда нужен доступ к **содержимому**:

- **Создание нового файла** (summary.md, tasks.md): `Write`
- **Чтение для анализа**: `Read`
- **Редактирование**: `StrReplace`
- **Удаление**: `Delete`
- **Поиск**: `Glob` / `Grep` (⚠️ ненадёжны для кириллических путей — см. секцию выше)

> **Правило**: `Read` + `Write` для «копирования» — только если файл МАЛЕНЬКИЙ (< 50 строк) и нужно его прочитать в том же flow. Для остального — Python shutil.

#### Создание директорий → НЕ НУЖЕН mkdir

Инструмент **Write** автоматически создаёт все промежуточные директории при записи файла. Для создания задачи или любой структуры папок достаточно вызвать `Write` на целевой файл — `mkdir` вообще не нужен.

> **ОПАСНОСТЬ**: `cmd /c mkdir` с кириллическим путём может **частично выполниться** даже при ошибке, оставив «призрачную» папку с обрезанным именем (например, `021-Упрощение` вместо `021-Упрощение корневой структуры для пользователя`). Эта папка не содержит файлов, но засоряет explorer.

#### НИКОГДА не использовать

- `Copy-Item` / `copy` / `Move-Item` / `move` с кириллическими путями в Shell
- `cmd /c` с кириллическими путями (в том числе `cmd /c mkdir` — даже при ошибке может создать мусорную директорию)
- `mkdir` через Shell для путей с кириллицей (использовать `Write` или Python `os.makedirs` с Unicode-эскейпами)
- Попытки «починить» encoding (`chcp 65001`, `$OutputEncoding`) — **НЕ помогают для записи** (`Copy-Item`, `Move-Item`)
- **НО**: `[Console]::OutputEncoding = [System.Text.Encoding]::UTF8; $OutputEncoding = [System.Text.Encoding]::UTF8;` **ПОМОГАЕТ для чтения** (`Get-ChildItem`, `Select-String`, пайплайны с кириллическим выводом). Добавлять в начало команд, которые **читают** или **перечисляют** файлы с кириллическими именами
- **Get-Content** для UTF-8 файлов: обязательно добавлять `-Encoding UTF8`, иначе кириллица внутри файла будет mojibake (PS по умолчанию читает как Windows-1251)
- **Python через Shell**: обязательно установить `$env:PYTHONIOENCODING = 'utf-8'; $env:PYTHONUTF8 = '1'` перед вызовом Python, иначе `print()` с кириллицей выдаёт мусор. Настройка `.vscode/settings.json` → `terminal.integrated.env.windows` покрывает это для интегрированного терминала, но Shell tool Cursor использует свой процесс
- `Read` + `Write` для копирования больших файлов (> 50 строк) — медленно, дорого, рискованно

### Типовой рецепт: сканирование директории с ограничительным `.gitignore`

Некоторые директории (например, `Инбокс/`) содержат `.gitignore` с паттерном `*`, чтобы не коммитить пользовательские файлы. Glob-инструмент Cursor уважает `.gitignore` и **не увидит файлы** в таких директориях. Использовать Python:

```python
python -c "import os; [print(f) for f in os.listdir(os.path.join('c:\\vaibe-os', '\u0418\u043d\u0431\u043e\u043a\u0441')) if f not in ('.gitkeep','.gitignore','README.md')]"
```

### Типовой рецепт: импорт файла из Инбокса

```python
# 1. Создать целевую директорию
python -c "import os; os.makedirs(os.path.join('c:\\vaibe-os', '\u041f\u0440\u043e\u0435\u043a\u0442\u044b', 'МойПроект', '\u0412\u0441\u0442\u0440\u0435\u0447\u0438', '\u0412\u0441\u0442\u0440\u0435\u0447\u0430 3. 10.02.2026 - \u041e\u0431\u0441\u0443\u0436\u0434\u0435\u043d\u0438\u0435'), exist_ok=True)"
```

## Общие правила для Shell-вызовов на Windows

1. **Одна команда = один вызов Shell.** Не цепочить через `&&`, `;`, `|` с побочными эффектами.
2. **Кириллица в путях** — НИКОГДА не передавать прямым текстом в shell-команды для файловых операций. Использовать Cursor-инструменты (Read/Write/Delete) или Python с Unicode-эскейпами (см. раздел выше).
3. **Staging теряется между вызовами Shell** — проверять через `git status` перед коммитом; если потерялся — повторить `git add`.
4. **Не использовать bash-измы**: `export`, `source`, `cat <<`, `$(...)`, `${}`.
5. **Кириллические пути ломаются в git** — консоль Windows (CP866) искажает UTF-8 пути. Если `git add`/`git hash-object` не находит файл с кириллицей, использовать Python: `subprocess.run(['git', 'add', '--', 'Путь/...'])` с Unicode-эскейпами.
6. **Git quotepath** — в `.git/config` должно быть `core.quotepath = false`, иначе `git status`/`git log` показывают кириллические пути как `\320\237\321\200\320\276\320\265\320\272\321\202\321\213/...` (octal escapes). Настройка уже применена в этом репозитории.
7. **Фантомные modified файлы** — если `git status` показывает modified, `git diff` пуст и `git add` не помогает — это конфликт регистра в именах (Windows case-insensitive, Linux case-sensitive). См. `.ai/rules/git-cross-platform.md`.

## Удаление директорий в PowerShell (частая ошибка)

- `rmdir /s /q ...` — это **cmd-стиль**. В PowerShell `rmdir` — алиас на `Remove-Item`, и параметры вида `/s /q` ломают команду.
- Использовать:

```powershell
Remove-Item -Recurse -Force ".tmp\\some-folder"
```

> Примечание: для путей с кириллицей действуют правила выше (предпочитать Cursor-инструменты или Python с Unicode-эскейпами).

## Гигиена: предотвращение мусорных папок

Shell-команды с кириллическими путями на Windows — главный источник «призрачных» директорий. Мусорные папки возникают двумя способами:

1. **Фрагментация**: `cmd /c mkdir "путь с пробелами"` при сбое кодировки интерпретирует пробелы как разделители пути, создавая отдельные папки из каждого слова (например, `для/`, `корневой/`, `структуры/`)
2. **Double-encoding**: shell-команды записывают UTF-8 байты как CP1252/CP866, создавая папки с mojibake-именами вместо нормальных кириллических

### Правила предотвращения

1. **Создание файлов и папок** — ТОЛЬКО через `Write` (автоматически создаёт все родительские директории) или Python `os.makedirs()` с Unicode-эскейпами. НИКОГДА через `mkdir`, `md`, `New-Item` в shell.
2. **Создание пустых папок** — через `Write` с файлом `.gitkeep` внутри.
3. **После любых файловых операций через Shell** — визуально проверить, не появились ли лишние папки в explorer или через Python-скрипт ниже.
4. **Если обнаружены мусорные папки** — удалить через Python `shutil.rmtree()` с Unicode-эскейпами, предварительно убедившись что папка пустая (0 файлов).

### Скрипт проверки (запускать при подозрении на мусор)

```python
python -c "import os; root=os.getcwd(); known={'.ai','.cursor','.claude','.git','.tmp','.venv','.vscode','repositories','tools','\u041f\u0440\u043e\u0435\u043a\u0442\u044b','\u0411\u0430\u0437\u0430 \u0437\u043d\u0430\u043d\u0438\u0439','\u041a\u043e\u043d\u0442\u0430\u043a\u0442\u044b','\u0418\u043d\u0431\u043e\u043a\u0441'}; [print(f'UNKNOWN [{sum(1 for _,_,f in os.walk(os.path.join(root,d)) for _ in f)} files]: {d}') for d in sorted(os.listdir(root)) if os.path.isdir(os.path.join(root,d)) and d not in known]"
```

## Экспорт готового результата в PDF (рекомендуемый путь)

Если пользователь просит артефакт в формате PDF, **не использовать** headless-браузер / внешние движки.

В репозитории уже есть конвертер:
- `tools/markdown_to_pdf/` (см. `tools/TOOLS_INDEX.md`)

Рецепт:
1) Создать/обновить Markdown-файл через Cursor-инструменты (в папке `results/` задачи).
2) Сгенерировать PDF через Python:

```powershell
python -m tools.markdown_to_pdf "input.md" -o "output.pdf"
```

Плюсы:
- не требует `pandoc`/LaTeX/wkhtmltopdf
- корректная кириллица (через системные Windows-шрифты)
