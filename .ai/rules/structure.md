# Структура файловой системы AI-OS

## Правила организации файлов и директорий

### Проекты
**Путь**: `Проекты/{NAME}/`

- Каждый проект хранится в отдельной подпапке с названием проекта
- Название папки проекта должно быть на русском языке
- Каждый проект имеет **видимость**: `командный` (по умолчанию) или `личный`
  - `командный` — трекается в Git, доступен всей команде
  - `личный` — исключён из Git через `.gitignore`, содержит приватные данные
  - Видимость фиксируется в `README.md` проекта и определяется при создании
  - При создании личного проекта — автоматически добавить `/Проекты/{NAME}/` в `.gitignore`
- Внутри папки проекта должны быть:
  - `README.md` - индексный файл проекта с метаданными, описанием и видимостью
  - `Задачи/` - подпапка для задач проекта
  - `Встречи/` - подпапка для транскрипций, самморизаций и задач с звонков
  - `Исходные материалы/` - подпапка для различных файлов о проекте
  - `База знаний/` - подпапка для накапливаемой объективной информации о проекте
  - `Планы/` - (опционально) snapshot-планы проекта
  - Другие файлы и папки проекта

**Пример**:
```
Проекты/
└── AI Операционная Система/
    ├── README.md
    ├── Задачи/
    ├── Встречи/
    ├── Исходные материалы/
    └── База знаний/
```

### Задачи
**Путь**: `Проекты/{NAME}/Задачи/{NUM}-{TITLE}/`

- Задачи хранятся в подпапке `Задачи/` внутри папки проекта
- Каждая задача в отдельной подпапке
- Формат названия папки задачи: `{NUM}-{TITLE}`
  - `{NUM}` - порядковый номер задачи (например, 001, 002)
  - `{TITLE}` - краткое название задачи на русском языке
- Внутри папки задачи должен быть файл `task.md` с описанием задачи
- Файл `task.md` ОБЯЗАТЕЛЬНО содержит секцию `## Статус` в конце файла

**Стандарт секции `## Статус` в task.md:**

Секция `## Статус` — единая точка правды о состоянии задачи. Всегда располагается **в конце** файла `task.md`.

```markdown
## Статус
- **Статус**: открыта | в процессе | выполнена | на холде
- **Завершена**: YYYY-MM-DD          ← только для выполненных
- **Версия результата**: v{N}        ← только для выполненных с файлами в results/
```

Значения `Версия результата`:
- `v1`, `v2`, ... — файлы в `results/v{N}/`
- `inline` — результат в самом `task.md` или структурные изменения (код, конфиги)
- `inline (описание)` — уточнение, где именно результат (например: `inline (код в repositories/)`)

**Пример**:
```
Проекты/
└── AI Операционная Система/
    └── Задачи/
        ├── 001-Проектирование системы/
        │   └── task.md
        └── 002-Реализация модулей/
            └── task.md
```

### Результаты задач (версионирование)
**Путь**: `Проекты/{NAME}/Задачи/{NUM}-{TITLE}/results/v{N}/`

- Результаты выполнения задачи хранятся в подпапке `results/` внутри папки задачи
- **Каждое выполнение** сохраняется в отдельную подпапку `v{N}/` (v1, v2, v3...)
- При повторном выполнении задачи — создаётся следующая версия, предыдущие не затираются
- `task.md` не дублируется в `results/` — он остаётся в корне задачи

**Алгоритм определения версии:**
1. Проверить `results/` на наличие подпапок `v{N}/`
2. Если нет ни одной — создать `v1/`
3. Если есть — найти максимальный N, создать `v{N+1}/`

**Пример (задача выполнена 3 раза):**
```
Проекты/
└── МойПроект/
    └── Задачи/
        └── 019-Макет визитной карточки/
            ├── task.md
            └── results/
                ├── v1/                    ← первое выполнение (Auto)
                │   └── business-card.html
                ├── v2/                    ← второе (Opus)
                │   └── business-card.html
                └── v3/                    ← третье (Composer 1.5)
                    └── business-card.html
```

**Важно:**
- Файлы напрямую в `results/` (без подпапки `v{N}/`) — legacy-формат, допустим, но при новом выполнении не затирать их, а создать `v1/` для legacy и `v2/` для нового результата
- Название файлов внутри `v{N}/` — на усмотрение задачи (латиница допустима для кода и HTML)

### Встречи
**Путь**: `Проекты/{NAME}/Встречи/Встреча {N}. {DATE:DD.MM.YYYY} - {TITLE}/`

- Встречи хранятся в подпапке `Встречи/` внутри папки проекта
- Каждая встреча в отдельной подпапке
- Формат названия папки встречи: `Встреча {N}. {DATE:DD.MM.YYYY} - {TITLE}`
  - `{N}` - порядковый номер встречи (например, 1, 2, 10)
  - `{DATE:DD.MM.YYYY}` - дата встречи в формате ДД.ММ.ГГГГ (например, 20.01.2026)
  - `{TITLE}` - краткое название встречи на русском языке
- Внутри папки встречи могут быть:
  - `transcript.md` - транскрипция встречи
  - `summary.md` - самморизация встречи
  - `tasks.md` - собранные задачи с звонка
  - Другие материалы встречи

**Пример**:
```
Проекты/
└── AI Операционная Система/
    └── Встречи/
        ├── Встреча 1. 20.01.2026 - Обсуждение архитектуры/
        │   ├── transcript.md
        │   ├── summary.md
        │   └── tasks.md
        └── Встреча 2. 25.01.2026 - Планирование спринта/
            ├── transcript.md
            ├── summary.md
            └── tasks.md
```

### Исходные материалы
**Путь**: `Проекты/{NAME}/Исходные материалы/`

- Исходные материалы хранятся в подпапке `Исходные материалы/` внутри папки проекта
- Содержит различные файлы о проекте: документы, спецификации, дизайн-макеты, схемы и т.д.
- Могут быть организованы в подпапки по типам материалов
- Названия файлов и подпапок на русском языке

**Пример**:
```
Проекты/
└── AI Операционная Система/
    └── Исходные материалы/
        ├── Спецификации/
        │   └── Техническое задание.md
        ├── Дизайн/
        │   └── Макеты интерфейса.png
        └── Схемы/
            └── Архитектура системы.drawio
```

### База знаний проекта
**Путь**: `Проекты/{NAME}/База знаний/`

- База знаний проекта хранится в подпапке `База знаний/` внутри папки проекта
- Содержит накапливаемую объективную информацию о проекте
- Может быть организована в подпапки по темам
- Названия файлов и подпапок на русском языке

**Пример**:
```
Проекты/
└── AI Операционная Система/
    └── База знаний/
        ├── Технологии/
        │   └── Используемый стек.md
        ├── Решения/
        │   └── Архитектурные решения.md
        └── Контекст/
            └── История проекта.md
```

### Skills, Knowledge и Ontology (универсальные)
**Путь**: `.ai/`

Универсальные playbooks, справочные материалы и философский фундамент, работающие в любой AI IDE (Cursor, Claude Code, OpenCode и др.).

**Иерархия документов**:
```
.ai/ONTOLOGY.md      — онтологический фундамент (природа и законы системы)
  └── MANIFESTO.md   — поведенческие принципы (выводятся из онтологии)
        └── router.md → skills  — механизмы исполнения
```

- `.ai/ONTOLOGY.md` — онтологический фундамент: пять законов, определяющих природу и развитие системы
- `.ai/router.md` — маршрутизатор: классификация задачи и выбор skill
- `.ai/skills/*.md` — actionable playbooks с процессом, форматом вывода и чек-листом качества
- `.ai/knowledge/*.md` — справочные материалы без actionable процесса

Формат skill-файла: YAML frontmatter (name, description, triggers) + секции Purpose, Procedure, Output format, Quality bar, Anti-patterns.

**Пример**:
```
.ai/
├── ONTOLOGY.md
├── router.md
├── skills/
│   ├── task-create.md
│   ├── task-execute.md
│   ├── evolve.md
│   └── ...
└── knowledge/
    ├── tech-stack-reference.md
    └── startup-resources-ru.md
```

### Планы (опционально)
**Путь**: `Проекты/{NAME}/Планы/`

- Snapshot-планы проекта: приоритеты, текущий спринт, статусы
- Формат имени файла: `План - {N} - {DD.MM.YYYY} - {TITLE}.md`
- Планы не редактируются "задним числом" — создаются новые версии
- Являются точкой интеграции (например, с YouTrack)

### Утилиты
**Путь**: `tools/`

- Директория для Python-утилит обслуживания vault
- Каждая утилита в отдельной подпапке: `tools/{tool_name}/`
- Вызов: `python -m tools.{tool_name}` или `python -m tools {command}`
- Индекс утилит: `tools/TOOLS_INDEX.md`
- Зависимости: `tools/{tool_name}/requirements.txt`

### База знаний пользователя
**Путь**: `База знаний/`

- База знаний пользователя ОС хранится в корневой папке `База знаний/`
- Содержит дополнительный контекст, который пользователь может выборочно добавлять в промпты
- Может содержать любые материалы: заметки, документацию, справочники и т.д.
- Названия файлов и подпапок на русском языке

**Пример**:
```
База знаний/
├── Заметки/
│   └── Идеи для улучшения.md
├── Справочники/
│   └── Команды Git.md
└── Документация/
    └── API компании.md
```

### Контакты
**Путь**: `Контакты/`

- Корневая папка со справочником персоналий, встречающихся в работе системы
- Является **источником истины** для имён, фамилий, ролей, алиасов и контекста по людям
- Один человек = один Markdown-файл
- Допустима организация по подпапкам категорий: `Команда/`, `ФРИИ/`, `Эксперты/`, `Клиенты/`, `Лиды/`
- Если в рабочих материалах встречаются сокращения, никнеймы или ошибочные варианты имени, они фиксируются в карточке контакта
- Для людей без подтверждённой фамилии использовать наиболее устойчивое имя и явно отмечать, что фамилия не указана

**Пример**:
```
Контакты/
├── README.md
├── Команда/
│   ├── Иван Иванов.md
│   └── Пётр Петров.md
├── Партнёры/
│   └── Мария.md
├── Эксперты/
│   └── Анна Сидорова.md
└── Лиды/
    └── Елена Козлова.md
```

### Репозитории
**Путь**: `repositories/`

- Директория для хранения git-репозиториев проектов с кодом
- Репозитории подключаются в виде git submodules
- Название папки на английском языке (`repositories`) для совместимости с git и инструментами разработки
- Внутри папки создаются подпапки для каждого подключённого репозитория

**Работа с submodules**:
- Добавление: `git submodule add <url> repositories/<name>`
- Обновление: `git submodule update --remote repositories/<name>`
- Клонирование проекта с submodules: `git clone --recurse-submodules <url>`
- Инициализация после клонирования: `git submodule update --init --recursive`

**Пример**:
```
repositories/
├── ai-os-backend/          # Backend-сервис системы
│   └── (содержимое репозитория)
├── ai-os-frontend/         # Frontend-приложение
│   └── (содержимое репозитория)
└── shared-libs/            # Общие библиотеки
    └── (содержимое репозитория)
```

### Инбокс
**Путь**: `Инбокс/`

- Инбокс - временная папка для хранения файлов, импортированных из сторонних систем
- Пользователь добавляет в Инбокс файлы различных форматов:
  - Скриншоты (PNG, JPG, etc.)
  - Транскрипты встреч (TXT, MD, DOCX, etc.)
  - Копии материалов из личных баз знаний
  - Другие файлы из внешних источников
- **ВАЖНО**: Файлы из Инбокса должны быть импортированы в структуру директорий ОС
- После импорта файлы удаляются из Инбокса или перемещаются в архив

**Правила импорта**:
- Используйте команду `/inbox-check` для обработки файлов из Инбокса
- Команда запрашивает контекст у пользователя о каждом файле
- Файлы импортируются в соответствующие директории согласно структуре ОС:
  - Материалы проектов → `Проекты/{NAME}/`
  - Пользовательские материалы → `База знаний/`
  - Документация задач → `Проекты/{NAME}/Задачи/{NUM}-{TITLE}/`
- После импорта файлы переименовываются согласно правилам именования ОС

**Пример**:
```
Инбокс/
├── Встреча по проекту 20.01.2026.txt
├── Скриншот интерфейса.png
└── Копия документации API.docx
```

## Правила именования

1. **Все названия папок и файлов на русском языке**
2. **Папки**: используйте заглавные буквы для первых букв слов (PascalCase для русских названий)
3. **Файлы**: используйте заглавные буквы для первых букв слов в названиях файлов
4. **Номера задач**: используйте формат с ведущими нулями (001, 002, 010, 100)
5. **ЗАПРЕЩЕНО менять только регистр** в имени папки/файла (например, `vAIbe-os` → `vAIbe-OS`). Это создаёт неразрешимые конфликты между Linux (case-sensitive) и Windows (case-insensitive). При необходимости — переименовывать через промежуточное имя в два коммита. Подробнее: `.ai/rules/git-cross-platform.md`

## Импорт файлов из Инбокса

Файлы из Инбокса должны быть импортированы в структуру директорий ОС с помощью команды `/inbox-check`. См. `.ai/skills/core/inbox-processing.md` для детального алгоритма импорта.

## Важные замечания

- **Строго соблюдайте структуру папок** - это критично для работы системы
- **Используйте только русский язык** для названий папок и файлов
- **Номера задач должны быть уникальными** в рамках проекта
- **При создании новых проектов или задач** всегда следуйте этой структуре
- **Новые знания** добавлять в `.ai/skills/domain/` (actionable) или `.ai/knowledge/` (reference)
- **Файлы из Инбокса должны быть обработаны** командой `/inbox-check` и импортированы в структуру ОС
