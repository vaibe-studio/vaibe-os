# Tools: переиспользование утилит из `tools/`

Цель: снизить "когнитивную деградацию" системы и не плодить дубликаты скриптов.

## Главное правило

Перед тем как:
- писать новый скрипт;
- предлагать пользователю "сделать руками" повторяемую операцию;
- добавлять новый `tools/<name>/...`;

сначала **проверь**, нет ли уже готового решения в:
- `tools/TOOLS_INDEX.md` (первый экран: что и как запускать)
- `tools/` (поиск по названиям и описаниям)

## Если подходящий инструмент уже есть

- **Переиспользуй** существующий инструмент.
- Если нужно — **расширь** его (добавь опцию/режим), а не создавай новый "почти такой же".
- После изменений — **обнови** `tools/TOOLS_INDEX.md` (команды, назначение, примечания).

## Если инструмента нет

Создавай инструмент так, чтобы его можно было переиспользовать:
- предпочитай структуру `tools/<tool>/` с:
  - `__init__.py`
  - `__main__.py` (чтобы работало `python -m tools.<tool> ...`)
  - `README.md` (как установить/запустить)
  - `requirements.txt` (точечные зависимости)
- добавь запись в `tools/TOOLS_INDEX.md`

## Проверка результатов поиска

Нельзя делать вывод "файл / папка отсутствует" только по одному инструменту поиска.

- Если `Glob` или другой поиск вернул пустой либо подозрительный результат, **обязательно** выполни fallback-проверку через другой инструмент: например, `rg`, чтение по ожидаемому пути или альтернативный scoped search.
- Если пользователь уже указал, что файл или папка существуют, сначала перепроверь несколькими способами, и только потом делай вывод о расхождении.
- При работе со структурой хранилища считай ложный отрицательный результат критичной ошибкой: лучше сделать лишнюю проверку, чем пропустить существующую задачу, проект или материал.

## Нотация команд (для ответов пользователю)

Всегда старайся давать **одну воспроизводимую команду**:

- если есть `__main__.py`: `python -m tools.<tool> ...`
- иначе: `python tools/<tool>/script.py ...`
