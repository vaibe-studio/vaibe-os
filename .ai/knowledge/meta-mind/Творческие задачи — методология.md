# Творческие задачи — методология meta-mind

> Как помогать пользователю решать творческие задачи: художественный перевод, адаптация текстов, создание контента под ограничения.

## Суть творческих задач

Творческие задачи отличаются от технических:
- **Нет единственного правильного ответа** — множество валидных решений
- **Критерии качества субъективны** — вкус, стиль, контекст
- **Ограничения могут быть мягкими или жёсткими** — от «сохрани смысл» до «уложись в ровно 8 слогов»
- **Итеративность** — результат улучшается через правки и обратную связь

## Типы творческих задач

| Тип | Примеры | Ключевые ограничения |
|-----|--------|----------------------|
| Художественный перевод | Песни, поэзия, проза | Смысл, стиль, ритм (если под музыку) |
| Адаптация под формат | Текст для соцсетей, подписи | Длина, тон, платформа |
| Создание с нуля (короткая форма) | Слоганы, названия, описания | Бриф, аудитория, тон |
| **Создание с нуля (песня)** | **Оригинальные песни по промпту** | **Форма, жанр, рифма, мелодика, стиль** |
| Переработка стиля | Академический → разговорный | Сохранение смысла, смена подачи |

## Алгоритм: художественный перевод (упрощённый)

*Когда не нужно укладываться в музыку — только смысл, структура и стилистика.*

### Шаг 1: Анализ оригинала
```
- Смысл-ядро: о чём текст на самом деле?
- Стиль-ядро: тон, регистр (разговорный/грубый/поэтичный)
- Структура: куплеты, припевы, повторы, ключевые фразы
- Культурные маркеры: сленг, отсылки, мат
```

### Шаг 2: Определение границ перевода
```
- Уровень грубости: сохранить / смягчить / усилить
- Регистр: разговорный русский / литературный / сленг
- Компрессия: русский часто длиннее — можно ли сжимать?
```

### Шаг 3: Варианты для ключевых строк
Для **хуков, повторов и самых ярких фраз** — предлагать 2–3 варианта перевода:
- с разным порядком слов
- с разной степенью грубости / буквальности
- с разной ритмикой (даже если перевод не под музыку — пользователь может знать ритм)

Формат:
```
Оригинал: I'ma fuck up my life
  а) Я проебал свою жизнь — прямой мат, ритмически плотно
  б) Я себе жизнь угроблю — грубо, но без мата
  в) Я просру свою жизнь — мат, другой оттенок (будущее время)
```

Пользователь выбирает, опираясь на своё чувство текста и (если есть) знание мелодии.

### Шаг 4: Черновик по блокам
- Переводить смысловыми блоками, не слово в слово
- Сначала ключевые фразы и хуки (из шага 3)
- Потом заполнять связки

### Шаг 5: Проверка стилистики
- Мат и грубость — на одном уровне по всей песне
- Разговорность — без «книжных» оборотов
- Образы — работают ли в русском? (метафоры, идиомы)

### Шаг 6: Таблица решений (опционально)
- Для спорных мест — явно зафиксировать выбор и обоснование
- Помогает при итерациях и обсуждении с пользователем

## Алгоритм: художественный перевод (под музыку)

*Когда текст должен петься на ту же мелодию.*

### Дополнительные ограничения
1. **Слоговая длина** — количество слогов на фразу
2. **Ударения** — ударный слог = музыкальный акцент
3. **Певучесть** — на длинных нотах — гласные, не стечения согласных
4. **Хуки** — повторы должны быть запоминающимися

### Методика
1. Разметить вокальную дорожку: слоги, акценты, паузы
2. Писать «скелет» под ритм (заглушки по слогам)
3. Заменять на смысловые слова с сохранением ударений
4. Компрессировать: русский длиннее — убирать лишнее
5. Проверять голосом: напеть 3–5 раз, править «спотыкающиеся» места

## Алгоритм: создание оригинальной песни

*Полный цикл от промпта пользователя до финального текста. Детальная методология, шаблоны промптов, примеры и упражнения — в `База знаний/Творчество/Написание песен — методология и приёмы.md`.*

### Отличие от перевода

При переводе — есть оригинал, задающий структуру, ритм, рифмосхему и смысл. При создании с нуля — всё это определяется автором (пользователем + meta-mind). Это даёт свободу, но требует больше решений на старте.

### Алгоритм

#### Шаг 1: Декомпозиция промпта
Разложить промпт пользователя на 6 осей: тема, настроение, жанр, аудитория, перспектива, ограничения. Если промпт абстрактный — задать уточняющие вопросы или предложить 3 конкретизации.

#### Шаг 2: Object Writing (опционально)
Brainstorm через 7 чувств (Pat Pattison): зрение, слух, обоняние, осязание, вкус, органическое чувство, кинестетика. Цель — сенсорный материал для образности.

#### Шаг 3: Смысловая карта
Эмоциональный маршрут (откуда → куда), ключевые образы (3–5), запретные зоны.

#### Шаг 4: Структура и карта ограничений
Выбрать форму (куплет-припев, сквозная, нарративная…). Определить, что происходит в каждой части. Зафиксировать ограничения (жёсткие / мягкие / пользовательские).

> **Критично**: определить форму **до** генерации текста. LLM ошибается в 80% слогов без явных ограничений (Song Form-Aware, 2024).

#### Шаг 5: Опорные рифмопары
Ключевые слова темы → рифмопары (подход Маршака: рифма — первый шаг, не последний). Использовать классификацию из `База знаний/Творчество/Рифмообразование — классификация и приёмы.md`.

#### Шаг 6: Генерация хука / припева
2–3 варианта разного типа (лозунг, парадокс, вопрос, образ, утверждение). Пользователь выбирает.

#### Шаг 7: Генерация куплетов и бриджа
По блокам, с вариантами для ключевых строк. Пользователь коллажирует и правит.

#### Шаг 8: Просодия-чек
- Все элементы поддерживают основную идею? (Pattison)
- Фонетика чиста? Гласные на длинных местах?
- Рифма не банальная, усиливает смысл?
- Хук запоминается?
- Эмоциональный маршрут: есть развитие?

#### Шаг 9: Итерация
Правки пользователя → новые варианты для конкретных мест → повторить шаги 6–8 по необходимости.

#### Шаг 10: Финал
Финальный текст + таблица решений + обратная связь (для мета-обучения).

### Три ключевых принципа (из исследований)

1. **LLM = генератор структуры и вариантов, не автор** — LLM отлично держит форму и тему, но проигрывает в креативности и эмоциях (POEMetric, ICLR 2026). Meta-mind генерирует каркас + материал, пользователь — душу.

2. **Человеческий ввод на старте критичен** — GPT-4 креативнее с человеческими заголовками, чем со своими (EMNLP 2024). Чем богаче промпт пользователя, тем лучше результат.

3. **Явные ограничения > расплывчатые запросы** — форма, рифмосхема, метр — задать конкретными числами до генерации (Song Form-Aware, 2024; Songs Across Borders, 2023).

## Принципы работы meta-mind с творческими задачами

### 1. Не навязывать единственный вариант
- Предлагать 2–3 варианта с разной степенью свободы
- Объяснять trade-off: «более буквально, но менее естественно»

### 2. Фиксировать решения
- Таблица «оригинал → решение → обоснование»
- Помогает пользователю понять логику и править точечно

### 3. Спрашивать о границах
- «Сохранить мат или смягчить?»
- «Приоритет — точность или естественность?»

### 4. Итеративность
- Первый черновик — не финал
- Готовность к правкам по конкретным строкам

### 5. Признавать субъективность
- «Это моя интерпретация; твой вкус может отличаться»
- Не защищать свой вариант — предлагать альтернативы

## Анти-паттерны

| ❌ Не делать | ✅ Делать вместо |
|-------------|------------------|
| Один «правильный» перевод | Несколько вариантов + обоснование |
| Игнорировать мат/грубость | Спросить уровень, сохранить стиль |
| Буквальный перевод идиом | Искать эквивалент в целевом языке |
| Защищать свой вариант | «Можем поменять, если не заходит» |
| Длинные пояснения без вариантов | Сначала вариант, потом при необходимости — пояснение |

## Исследования: LLM и художественный перевод (2023–2025)

> Обзор академических работ с практическими выводами для meta-mind.

### 1. EAPMT — «Объясни, потом переведи» (Wang et al., 2024)

**Источник**: [What is the Best Way for ChatGPT to Translate Poetry?](https://arxiv.org/abs/2406.03450) — ACL 2024

**Суть**: Вместо прямого перевода стихотворения — сначала попросить LLM **объяснить смысл** каждой строфы на исходном языке, затем использовать это объяснение как контекст для перевода.

**Результат**: Превосходит прямой перевод ChatGPT и онлайн-переводчики. Подтверждено панелью профессиональных поэтов + GPT-4.

**Применение для meta-mind**:
- Перед переводом — создать «смысловую карту»: что означает каждая строфа, какие образы, какой подтекст
- Подавать эту карту как контекст в промпт перевода
- Особенно полезно для сложных метафор и культурных отсылок

### 2. Songs Across Borders — ограничения как промпты (Ou et al., 2023)

**Источник**: [Songs Across Borders: Singable and Controllable Neural Lyric Translation](https://aclanthology.org/2023.acl-long.27/) — ACL 2023 | [GitHub](https://github.com/Sonata165/ControllableLyricTranslation)

**Суть**: Перевод песен формализован как **задача с ограничениями**. Теория переводоведения конвертируется в конкретные промпт-ограничения: количество слогов, схема рифм, границы слов.

**Результат** (EN→ZH): 99.85% точность длины, 99% точность рифмы, 95.52% recall границ слов.

**Применение для meta-mind**:
- Явно указывать ограничения в промпте: «ровно N слогов», «рифма AABB», «ударение на 3-й слог»
- Чем конкретнее ограничения — тем лучше результат (расплывчатые запросы дают банальный текст)
- Для перевода под музыку — разметить ограничения по каждой строке, затем подать в промпт

### 3. Sing it, Narrate it — качество + певучесть (Ye et al., 2024)

**Источник**: [Sing it, Narrate it: Quality Musical Lyrics Translation](https://arxiv.org/abs/2410.22066) — EMNLP 2024

**Суть**: Двухстадийный подход: (1) reward model для автоматической оценки качества перевода, (2) inference-time оптимизация для целых песен (не строка-за-строкой).

**Применение для meta-mind**:
- Переводить **целую песню / секцию**, а не строку-за-строкой — сохраняет связность
- После черновика — оценивать качество отдельно по осям: смысл, стиль, певучесть
- Итеративно улучшать по слабой оси

### 4. The Translator's Canvas — LLM как инструмент, не замена (Resende & Hadley, 2024)

**Источник**: [The Translator's Canvas](https://aclanthology.org/2024.amta-research.16/) — AMTA 2024

**Суть**: LLM работает лучше всего как **инструмент-помощник** для переводчика (генерация вариантов, подсказки рифм, альтернативные формулировки), а не как автономный переводчик. Пост-редактирование LLM-выхода часто сложнее, чем перевод с нуля.

**Применение для meta-mind**:
- Роль LLM = **генератор вариантов**, финальное решение — за пользователем
- Не пытаться выдать «готовый перевод» — давать материал для выбора
- Предлагать рифмы, синонимы, порядки слов — пользователь собирает финал

### 5. LITRANSPROQA — оценка литературного перевода (2025)

**Источник**: [EMNLP 2025](https://aclanthology.org/2025.emnlp-main.1482.pdf)

**Суть**: Новая метрика оценки литературного перевода через профессиональные вопросы. Фокус на: литературные приёмы, культурный контекст, авторский голос. Стандартные метрики (BLEU и др.) переоценивают качество машинного перевода.

**Применение для meta-mind**:
- Оценивать перевод не по «похожести на оригинал», а по: (1) сохранение приёмов, (2) культурная адаптация, (3) голос автора
- Механическая точность ≠ художественное качество

## Практические выводы для промптинга

На основе исследований — чек-лист для meta-mind при творческом переводе:

1. **Сначала объясни, потом переводи** (EAPMT): создать смысловую карту оригинала
2. **Явные ограничения в промпте** (Songs Across Borders): слоги, рифмы, ударения — конкретные числа
3. **Целая секция, не строка** (Sing it, Narrate it): переводить блоками для связности
4. **LLM = генератор вариантов** (Translator's Canvas): не «готовый перевод», а материал для выбора
5. **Оценка по 3 осям** (LITRANSPROQA): приёмы + культура + голос, не BLEU-score

## Архитектура мета-обучения

> Не только «как переводить», но и «как meta-mind накапливает мастерство через практику».

### Цикл обучения

```
Практика → Захват → Обобщение → Интеграция → Применение
    ↑                                              |
    └──────────────────────────────────────────────┘
```

**Практика** — выполнение творческой задачи вместе с пользователем: генерация вариантов, получение выбора и правок.

**Захват** — фиксация решений пользователя: что выбрал, что отверг, что переписал с нуля, и почему (если объяснил).

**Обобщение** — поиск повторяющихся паттернов. «Пользователь всегда предпочитает X над Y в контексте Z» → формулировка правила.

**Интеграция** — сохранение через `/evolve` в базу знаний: обновление дефолтных параметров, расширение примеров, уточнение алгоритмов.

**Применение** — использование накопленного в следующей сессии. Явное указание: «Ориентируюсь на твой стиль из перевода Joji — скорректируй, если этот текст другой».

### Что фиксировать после каждой творческой сессии

| Сигнал | Пример | Как использовать |
|--------|--------|------------------|
| Выбор варианта | Выбрал (а) из трёх | Увеличить вес подобных вариантов |
| Правка текста | Поменял «проебал» на «просрал» | Скорректировать дефолт грубости |
| Отказ от варианта | «Слишком литературно» | Снизить формальность |
| Объяснение | «Мне важнее ритм, чем точность» | Обновить приоритеты ограничений |
| Переписывание с нуля | Свой вариант строки | Анализировать разницу, извлечь паттерн |

### Принцип постепенной калибровки

meta-mind **не пытается угадать** всё с первой сессии. Калибровка происходит итеративно:

1. **Первый перевод** — работа на дефолтах + активный запрос предпочтений
2. **Второй перевод** — использование паттернов из первого, меньше вопросов
3. **N-й перевод** — стабильный профиль, вопросы только при новом жанре/стиле

Ключевое: LLM не имеет памяти между сессиями → профиль должен быть **явно записан** в базе знаний и **явно загружен** в контекст.

### Хранение

- Примеры переводов: `База знаний/Творчество/{Артист} - {Песня} (перевод).md`
- Разбор примеров: `База знаний/Творчество/Примеры переводов — разбор.md`
- Референсная база: `База знаний/Творчество/Референсная база переводов.md`
- Паттерны и правила: обновление этого файла через `/evolve`

## Унифицированный фреймворк: творческая задача с ограничениями

> Любая творческая задача — генерация текста в пространстве ограничений. Единый фреймворк вместо отдельных алгоритмов.

### Модель

```
Входы:
  ├── Исходный материал (оригинал / бриф / задание)
  ├── Ограничения (жёсткие + мягкие + пользовательские)
  └── Контекст (жанр, аудитория, платформа, история)

Процесс:
  1. Анализ → Смысловая карта + Карта ограничений
  2. Генерация → Варианты (2–3 на ключевых точках)
  3. Оценка → По осям ограничений
  4. Итерация → Правки пользователя → Новые варианты

Выход:
  ├── Финальный текст
  ├── Таблица решений (для воспроизводимости)
  └── Обратная связь (для мета-обучения)
```

### Типы ограничений

| Тип | Примеры | Нарушение = |
|-----|---------|-------------|
| **Жёсткие** | Кол-во слогов, длина, формат | Брак (не пригоден) |
| **Мягкие** | Стиль, тон, образность | Снижение качества |
| **Пользовательские** | «Без мата», «как у Летова», «ритм важнее» | Несоответствие ожиданиям |

### Карта ограничений по типам задач

| Задача | Жёсткие | Мягкие | Пользовательские |
|--------|---------|--------|------------------|
| Перевод песни (под музыку) | Слоги, ударения, рифмосхема | Смысл, образы, тон | Уровень грубости, стиль |
| Перевод стихотворения | Размер, рифмосхема (если сохраняем) | Стиль автора, эпоха | Модернизировать или нет |
| Свободный перевод (текст) | Нет | Смысл, стиль, тон | Уровень свободы |
| **Оригинальная песня** | **Форма песни, слоги (если под мелодию)** | **Жанр, образность, рифма, мелодика** | **Стиль артиста, регистр, тема** |
| **Оригинальная песня (под мелодию)** | **Слоги, ударения, форма** | **Жанр, рифма, тон** | **Стиль, тема, регистр** |
| Рерайт для соцсетей | Длина, платформа | Вовлечённость, тон | Бренд-голос |
| Слоган / нейминг | Длина, запоминаемость | Ассоциации, ритм | Целевая аудитория |
| Стилизация | Стиль-образец | Естественность | Конкретный автор |

### Алгоритм (универсальный, расширяет базовые)

#### Шаг 0: Карта ограничений
Перед началом работы — явно перечислить все ограничения и их приоритеты.

```
Задача: перевод песни Joji — Yeah Right
Жёсткие: нет (перевод не под музыку)
Мягкие: тон (цинизм, отстранённость), мат на уровне оригинала
Пользовательские: разговорный русский, без литературщины
Приоритет: стиль > ритм > рифма > точность
```

> **Примечание о рифме**: при любой задаче, связанной с поэзией или песнями, явно указать приоритет рифмы в иерархии ограничений. Использовать классификацию и чек-лист из `База знаний/Творчество/Рифмообразование — классификация и приёмы.md`. Ключевой вопрос: «Стоит ли сохранять/создавать рифму, и если да — какого типа?»

#### Шаг 1: Смысловая карта (EAPMT)
Перед генерацией — объяснить себе каждый блок:
- Буквальный смысл
- Подтекст / эмоция
- Культурные отсылки
- Тональные маркеры

#### Шаг 2: Генерация вариантов
- 2–3 варианта для **ключевых точек** (хуки, повторы, яркие фразы)
- Полный черновик для связующего текста
- Переводить **блоками** (секция / строфа), не строка-за-строкой

#### Шаг 3: Оценка по осям
Для каждого варианта:
- Смысловая точность (1–5)
- Стилистическое соответствие (1–5)
- Соответствие жёстким ограничениям (да/нет)
- Певучесть / ритмика (1–5, если релевантно)
- Качество рифмы (1–5, если релевантно): оригинальность, тип, семантическая связь (подробнее → `База знаний/Творчество/Рифмообразование — классификация и приёмы.md`)

#### Шаг 4: Презентация и итерация
- Показать варианты с обоснованием trade-off
- Принять выбор/правку пользователя
- Зафиксировать решение в таблице
- При необходимости — вернуться к шагу 2

## Шаблоны промптов для творческих задач

> Конкретные шаблоны на основе исследований EAPMT (Wang et al., 2024) и Songs Across Borders (Ou et al., 2023).

### Шаблон 1: EAPMT — перевод через объяснение

Двухэтапный подход: сначала смысловая карта, потом перевод. Особенно эффективен для сложных метафор и культурных отсылок.

```
[Системный контекст]
Ты — помощник для художественного перевода. Используй двухэтапный подход:
1. Объясни смысл каждого блока оригинала
2. Переведи, опираясь на своё объяснение

[Промпт]
Переведи на русский:

{текст}

Этап 1 — Смысловая карта:
Для каждого блока (куплет/строфа/абзац):
- Буквальный смысл
- Скрытый смысл / подтекст
- Тон и эмоция
- Культурные отсылки (если есть)

Этап 2 — Перевод:
На основе смысловой карты создай художественный перевод.
Стиль: {описание стиля}
Ограничения: {список}

Для ключевых строк — дай 2–3 варианта:
(а) ближе к оригиналу
(б) свободнее, но с сохранением тона
(в) с другим ритмом/грубостью
```

### Шаблон 2: Перевод под музыку (constraint-driven)

Ограничения как параметры. Для каждой строки — конкретные числа.

```
[Системный контекст]
Ты переводишь текст песни для исполнения на ту же мелодию.
Строго соблюдай ограничения по слогам и ударениям.

[Промпт]
Строка оригинала: {строка}
Слоги: {N}
Ударения на позициях: {список}
Рифма с: {предыдущая строка / слово}
Смысл: {пересказ одним предложением}
Тон: {описание}

Дай 3 варианта. Для каждого укажи:
- Текст
- Кол-во слогов (проверка)
- Позиции ударений (проверка)
- Рифма (проверка)
- Отклонения от смысла (если есть)
```

### Шаблон 3: Адаптация текста под формат

```
[Промпт]
Исходный текст:
{текст}

Целевой формат: {пост Telegram / tweet / описание YouTube / etc.}

Ограничения:
- Максимум: {N символов}
- Тон: {описание}
- Обязательно включить: {ключевые элементы}
- Исключить: {что убрать}

Аудитория: {описание}

Дай 2–3 варианта адаптации с разной степенью свободы.
```

### Шаблон 4: Стилизация / рерайт

```
[Промпт]
Исходный текст:
{текст}

Целевой стиль: {описание или пример — «как пишет X», «разговорный блог», etc.}

Сохранить обязательно: {фактическое содержание / ключевые термины}
Можно менять: {структуру / порядок / метафоры}

Дай 2 варианта:
(а) ближе к оригиналу по структуре
(б) свободнее, ближе к целевому стилю
```

### Использование шаблонов

Шаблоны — **отправная точка**, не догма. Адаптировать под конкретную задачу:
- Убрать лишние параметры для простых задач
- Добавить специфические ограничения для сложных
- Комбинировать шаблоны (EAPMT + constraint-driven для перевода песни)

## Механизм обратной связи

> Как фиксировать правки пользователя и интегрировать в будущие переводы.

### Типы сигналов

| Сигнал | Что произошло | Сила сигнала |
|--------|---------------|--------------|
| **Выбор** | Пользователь выбрал вариант (а) из трёх | Средняя — предпочтение, но не правило |
| **Правка** | Изменил текст (заменил слово/фразу) | Высокая — конкретное несогласие |
| **Переписывание** | Написал свой вариант с нуля | Очень высокая — образец целевого стиля |
| **Объяснение** | «Тут нужно грубее» / «Не тот ритм» | Очень высокая — явное правило |
| **Отказ** | «Не то, переделай» без деталей | Низкая — нужно уточнить |

### Формат фиксации

В примечаниях к каждому переводу — блок обратной связи:

```markdown
## Обратная связь

### Выборы
- "I'ma fuck up my life": (а) «Я проебал свою жизнь» ← прямой мат, ритм
- "She don't care if I die": (б) «Ей похуй, сдохну я» ← максимальная грубость

### Правки
- Добавил «ясно» для ритма: «Никогда не будем вместе, ясно»

### Извлечённые паттерны
- Предпочитает прямой мат → литературных эвфемизмов
- Ритм > точность
- Добавляет слова-заполнители для ритмического баланса
```

### Интеграция в будущие сессии

1. **Перед началом** — загрузить предыдущие примеры пользователя в контекст
2. **Применить паттерны** — как дефолтные настройки (грубость, приоритеты, стиль)
3. **Сообщить явно** — «Ориентируюсь на стиль из перевода Joji. Скорректируй, если этот текст требует другого подхода»
4. **Зафиксировать через /evolve** — обобщённые паттерны → в методологию

### Накопление профиля пользователя

С каждым переводом профиль уточняется:

```
Профиль: творческие переводы
├── Грубость: высокая (мат = ок)
├── Приоритет: ритм > стиль > смысл > рифма
├── Регистр: разговорный русский, без литературщины
├── Паттерны:
│   ├── Прямой перевод мата, не эвфемизмы
│   ├── Добавление слов для ритма — допустимо
│   └── Варианты обязательны для ключевых строк
└── Антипаттерны:
    ├── Книжные обороты
    └── Защита единственного варианта
```

LLM не имеет памяти между сессиями → профиль загружается через файлы базы знаний и контекст в промпте.

## Работа с рифмой

> Рифма — отдельная ось мастерства, требующая специальных знаний. Подробная классификация, приёмы мастеров, промпт-шаблоны и чек-листы вынесены в отдельный документ.

**Ссылка**: `База знаний/Творчество/Рифмообразование — классификация и приёмы.md`

### Ключевые принципы (краткая выжимка)

1. **Рифма — когнитивный инструмент**, не декорация: облегчает восприятие, усиливает запоминание, создаёт эстетическое удовольствие (подтверждено нейроисследованиями)
2. **Рифма как смысловой акцент** (Пушкин): рифмующиеся слова должны нести ключевую семантику
3. **Неожиданность** (Маяковский): предсказуемая рифма = потерянная возможность; составные и мультисложные рифмы усиливают текст
4. **Рифма как точка входа** (Маршак): при переводе с рифмой — начинать с поиска опорных рифмопар, строить текст вокруг них
5. **Рифма соединяет несовместимое** (Бродский): столкновение миров через рифму порождает новый смысл
6. **Рэп-расширение**: внутренние, мультисложные и артикуляционные рифмы — легитимные инструменты, не ограничиваться концом строки

### Когда работать с рифмой

| Ситуация | Приоритет рифмы | Действие |
|----------|----------------|----------|
| Перевод песни под музыку | Средний (после слогов и ударений) | Сохранять в припевах, ослаблять в куплетах |
| Перевод стихотворения | Высокий (если в оригинале есть) | Использовать подход Маршака: рифмопары → текст |
| Свободный перевод | Низкий | Проверять на случайные бонусные рифмы |
| Создание оригинального текста | По заданию | Явно определить схему до начала работы |
| Рэп/слэм | Критичный | Мультисложные, внутренние, цепные |
